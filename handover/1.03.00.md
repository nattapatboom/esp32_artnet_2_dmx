# Handover เอกสารส่งมอบงาน v1.3.0: Art-Net Node Device (WT32-ETH01)

เอกสารฉบับนี้สรุปความคืบหน้าในเวอร์ชัน **v1.3.0** ซึ่งเป็นผลลัพธ์จากการดำเนินการเสร็จสิ้นใน **Phase 3.5 (Quick Fixes)** เพื่อปรับปรุงประสบการณ์ใช้งาน (UX) และขยายขีดความสามารถของ ESP-NOW Master 

---

## 1. การปรับปรุงที่สำเร็จในเวอร์ชัน v1.3.0

### 1.1 `web_pages.h` — การตรวจสอบความขัดแย้งของ GPIO (GPIO Conflict Detection)
- **ปัญหา**: ผู้ใช้สามารถกำหนด GPIO pin ซ้ำกันในโหมดการตั้งค่า Output (เช่น ให้ DMX และ LED ใช้ GPIO 4 ทั้งคู่) ซึ่งจะทำให้สัญญาณชนกันและระบบทำงานผิดปกติ
- **การแก้ไข**: เพิ่มโค้ด JavaScript ฝั่ง Web UI ตรวจสอบการเลือก `no_pin` ก่อนกด Add หรือ Update หาก pin นั้นถูกเลือกไว้ใน Output ช่องอื่นแล้ว จะแจ้งเตือน `alert()` เพื่อบล็อกการดำเนินการ

### 1.2 `main.cpp` — การระบุบอร์ด Slave อัตโนมัติ (Slave AP SSID Suffix)
- **ปัญหา**: เมื่อมีบอร์ด ESP32 ที่เป็น Slave หลายตัว เปิด AP Mode พร้อมกันในชื่อ "ESP32-ArtNet-Setup" ทำให้แยกไม่ออกว่าบอร์ดไหนเป็นบอร์ดไหน
- **การแก้ไข**: หากผู้ใช้ตั้งค่าโหมดเป็น `MODE_ESPNOW_SLAVE` และ SSID ยังคงเป็นค่าเริ่มต้น (`ESP32-ArtNet-Setup`) ระบบจะต่อท้าย SSID ด้วยเลข 4 ตัวสุดท้ายของ MAC Address อัตโนมัติ (เช่น `ESP32-ArtNet-Setup-A1B2`) แบบ on-the-fly

### 1.3 `web_pages.h` — ลบช่อง Art-Net Universe ที่ซ้ำซ้อน
- **ปัญหา**: ในแท็บ Network มีการให้กรอก `artnet_universe` แต่ละ Output Channel มีค่า `start_universe` ของตัวเองอยู่แล้ว ทำให้เกิดความสับสนและทับซ้อนทางคอนเซปต์
- **การแก้ไข**: เอา input field ของ Universe ออกจาก Network UI แล้ว (เหลือเฉพาะ DMX Output FPS) 

### 1.4 `espnow_control.h` & `main.cpp` — ESP-NOW Multi-Peer MAC List
- **ปัญหา**: ระบบ ESP-NOW ฝั่ง Master รองรับเพียงแค่ 1 peer MAC Address หากมี Slave หลายตัว จะต้องใช้โหมด Broadcast ซึ่งจะส่งผลให้ Master บอร์ดอื่นๆ ในพื้นที่ใกล้เคียงรับแพ็กเก็ตข้ามระบบ (crosstalk) 
- **การแก้ไข**: 
  - เพิ่ม API `GET/POST /api/espnow-peers` อ่าน/เขียนอาร์เรย์ MAC Address ในรูปแบบ JSON (`/espnow_peers.json`)
  - หน้าต่าง ESP-NOW ใน UI เปลี่ยนเป็นตารางให้เพิ่ม/ลด MAC Address ของ Slave หลายๆ ตัวได้ (เหมือนกับของ WiZ Bulb)
  - โมดูล ESP-NOW Control เปลี่ยนไปใช้ระบบ `std::vector` ลงทะเบียน peer ย่อยทุกตัวที่อยู่ใน List และส่งข้อมูล (Unicast) วนลูปหาทีละตัว หาก List ว่าง จะใช้ระบบ Broadcast (FF:FF:FF:FF:FF:FF) โดยอัตโนมัติ

## 2. รายการไฟล์ที่มีการเปลี่ยนแปลง

1. **`include/web_pages.h`**: เพิ่ม JavaScript สำหรับตรวจจับ GPIO ซ้ำ, สร้างตาราง ESP-NOW peer management, ลบ UI เดิมที่ทิ้งไป
2. **`src/main.cpp`**: อัปเดตตรรกะ `setupNetwork()` สำหรับ AP SSID Suffix, เพิ่ม Web API endpoints `/api/espnow-peers`
3. **`include/espnow_control.h`**: ทำระบบ Load Peer list จาก LittleFS ด้วย `ArduinoJson`, ลงทะเบียน MAC หลายตัว, และเพิ่ม Loop ส่ง Unicast เข้าหา Slave ทีละตัว
4. **`AI_CONTEXT.md`**: อัปเดตสถานะเช็คลิสต์ Phase 3.5 ว่าเป็น "เสร็จแล้ว"

## 4. สิ่งที่ต้องทำต่อในเวอร์ชันถัดไป (Next Steps)

เข้าสู่ **Phase 4 — Recovery Mode & WiZ Merge**
1. **Recovery Mode**: จับสัญญาณเวลากดปุ่ม Reset บนบอร์ดรัวๆ (3 ครั้งใน 3 วินาที) ให้เข้าโหมด Safe Mode โดยใช้ `RTC_DATA_ATTR` เพื่อนับจำนวนครั้ง Boot 
2. **WiZ Bulb in Output Channels**: สร้างประเภทช่อง Output ที่ 4 คือ `CHAN_TYPE_WIZ` นำหลอด WiZ ไปใส่รวมกันใน Output tab แทนที่แท็บแยก ช่วยรวมระบบคอนฟิกเข้าที่เดียว
