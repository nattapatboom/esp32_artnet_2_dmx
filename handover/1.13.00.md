### 3.1 DC Motor Hybrid Mapping

**อ้างอิง:** handover_v1.12.0 ข้อ 3.3 (Hybrid Output Mapping), ข้อ 18.1 ลำดับที่ 1

**ปัญหาปัจจุบัน:** DC Motor (type 6) รองรับแค่ ESP32 GPIO (source=0) และ PCA9685 (source=1) เท่านั้น ไม่มีโค้ดสำหรับใช้ digital expander (source=2-4) สำหรับขา DIR/IN

**แนวทางแก้ไข:**
- เพิ่มฟิลด์ขาเสริมสำหรับ DC Motor คล้ายกับที่ทำกับ Stepper (pin2_source, pin3_source)
- PWM pin ยังคงอยู่บน ESP32 GPIO หรือ PCA9685
- DIR/IN1/IN2 pins สามารถย้ายไป digital expander (MCP23017, TCA9555, PCF857x)
- เพิ่ม Web UI สำหรับเลือก source/address/channel ของขา DIR/IN
- ใช้ `writeHybridPin()` ที่มีอยู่แล้วใน motion_control.h (ปัจจุบันใช้เฉพาะ Stepper)

**ไฟล์ที่ต้องแก้ไข:**
- `include/motion_control.h` — ขยาย DC Motor `begin()` ให้รองรับ expander source
- `include/output_control.h` — เพิ่ม pin2_source/pin3_source/pin4_source สำหรับ DC Motor hybrid pin (ถ้ายังไม่มี field)
- `include/web_pages.h` — เพิ่ม UI สำหรับเลือก source/address/channel ของ DC Motor hybrid pins
- `src/main.cpp` — อัปเดต API validation ให้รองรับ field ใหม่

### 3.3 Homing Input on Digital Expander

**อ้างอิง:** handover_v1.12.0 ข้อ 3.3 (Homing Limit Switches), ข้อ 18.1 ลำดับที่ 3

**ปัญหาปัจจุบัน:** `pin4` ของ Stepper ต่อกับ homing sensor โดยตรงที่ ESP32 GPIO (`digitalRead(ch.pin4)`) ไม่มี `pin4_source` สำหรับเลือก expander

**แนวทางแก้ไข:**
- เพิ่มฟิลด์ `pin4_source`, `pin4_addr`, `pin4_channel` สำหรับ homing input
- ที่ support: MCP23017 (มี interrupt pin ด้วย), TCA9555, PCF857x
- อ่านค่าผ่าน `DigitalExpanderManager::digitalRead()` แทน `digitalRead()` ของ ESP32
- เริ่มต้นด้วย polling ก่อน (วนอ่านค่าทุก loop) แล้วค่อยเพิ่ม interrupt option ทีหลัง

**ไฟล์ที่ต้องแก้ไข:**
- `include/output_control.h` — เพิ่ม pin4_source/pin4_addr/pin4_channel
- `include/motion_control.h` — แก้ไข homing routine ให้อ่านจาก expander
- `include/i2c_gpio_expander.h` — เพิ่ม method digitalRead() (ถ้ายังไม่มี)
- `include/web_pages.h` — เพิ่ม UI สำหรับเลือก homing sensor source

### 3.4 LED Strip 400kHz Protocol Selection (WS2811 Low-Speed)

**อ้างอิง:** handover_v1.12.0 ข้อ 4.1 (LED Strip Protocol Classification)

**ปัญหาปัจจุบัน:** LED Strip รองรับเฉพาะกลุ่ม 800kHz (WS2812B, WS2813, SK6812, GS1903, UCS1903) ด้วย `NeoEsp32RmtXWs2812xMethod` ไม่มีตัวเลือกสำหรับ WS2811 รุ่นเก่าที่ใช้ความเร็ว 400kHz ซึ่งต้องใช้พัลส์ที่กว้างกว่า หากต่อกับไฟ 12V/24V WS2811 บางรุ่นจะไม่สามารถขับข้อมูลได้

**แนวทางแก้ไข:**
- เพิ่มตัวเลือก `LED_CHIP_TYPE` หรือ `mc_mode` สำหรับ LED Strip (type 0):
  - `0` = WS2812x 800kHz (ค่าเริ่มต้น)
  - `1` = WS2811 400kHz
- เมื่อเลือก 400kHz: เปลี่ยน method template เป็น `NeoEsp32RmtXWs2811Method` แทน `NeoEsp32RmtXWs2812xMethod`
- เพิ่ม Dropdown ใน Web UI สำหรับเลือก chip type (แสดงเฉพาะตอนเลือก type = LED Strip)
- ค่าเริ่มต้นเป็น 800kHz เพื่อคง backward compatibility

**ไฟล์ที่ต้องแก้ไข:**
- `include/output_control.h` — เพิ่ม method factory ที่เลือก 800kHz หรือ 400kHz ตาม config
- `include/web_pages.h` — เพิ่ม Dropdown chip type ใน LED Strip config section, ส่งค่าใน saveOutput

### 3.6 Direct-Drive 7-Segment via Digital Expander

**อ้างอิง:** handover_v1.12.0 ข้อ 8.1 (7-Pin / 8-Pin Direct Drive Layout), ข้อ 8.2 (A-Z Font Table)

**ปัญหาปัจจุบัน:** 7-Segment (type 13) รองรับเฉพาะ TM1637 protocol (bit-banged CLK/DIO) บน ESP32 GPIO เท่านั้น ยังไม่มีโหมด direct-drive ที่ต่อ segment A-G/DP เข้ากับ expander pins

**แนวทางแก้ไข:**
- เพิ่ม `mc_mode` ใหม่สำหรับ 7-Segment direct drive:
  - `2` = Direct Drive 7-pin (no DP)
  - `3` = Direct Drive 8-pin (with DP)
- ใน direct drive mode:
  - pins 1-8 (A-G, DP) ใช้ expander channels เรียงลำดับ (address + channel)
  - หรือใช้ field `pca_channel` - `pca_channel+7` สำหรับช่องสัญญาณ
- ใช้ font table ASCII-to-segment จาก handover_v1.12.0 ข้อ 8.2
- เปลี่ยนการ render จาก `TM1637Driver::displayRaw()` เป็น expander pin writes

**ไฟล์ที่ต้องแก้ไข:**
- `include/motion_control.h` — เพิ่ม direct drive mode ใน 7-segment section
- `include/output_control.h` — เพิ่ม mode/physical pin fields สำหรับ direct drive
- `include/i2c_gpio_expander.h` — ยืนยันว่า `digitalWrite()` ทำงานแบบ batch หรือ single channel
- `include/web_pages.h` — เพิ่ม UI สำหรับ direct drive config

### 3.8 I2C Display (OLED/LCD) Implementation

**อ้างอิง:** handover_v1.12.0 ข้อ 5 (I2C Display Implementation & Diagnostics Plan)

**ปัญหาปัจจุบัน:** ไม่มีไดรเวอร์หรือโค้ดสำหรับ OLED (SSD1306/SH1106) หรือ Character LCD (PCF8574 backpack) ใน codebase

**แนวทางแก้ไข:**
- เลือก library ขนาดเล็ก non-blocking:
  - OLED: `SSD1306Ascii` (เฉพาะ text, ไม่ต้อง full frame buffer)
  - หรือ `U8g2` (graphics, แต่ใช้ RAM มากกว่า)
  - LCD: `LiquidCrystal_I2C` (น้ำหนักเบา)
- แสดง 4 บรรทัดตามแผน:
  1. IP Address
  2. Network status (Link Up/Down)
  3. Art-Net universe range
  4. Output FPS / Temp
- Refresh rate 2-5 Hz (non-blocking)
- I2C address auto-detect หรือตั้งค่าผ่าน Web UI

**ไฟล์ที่ต้องแก้ไข:**
- `platformio.ini` — เพิ่ม dependencies (SSD1306Ascii หรือ U8g2 + LiquidCrystal_I2C)
- `include/` — ไฟล์ header ใหม่สำหรับ display driver
- `src/main.cpp` — initialize display ใน networkTask, update display status
- `include/web_pages.h` — เพิ่ม settings สำหรับ enable display, select type, I2C address

### 3.9 Dynamic Output Scoring System

**อ้างอิง:** handover_v1.12.0 ข้อ 7 (Dynamic Output Scoring System Plan)

**ปัญหาปัจจุบัน:** ระบบยังไม่มี Resource Point scoring การ validate output limits ทำแบบ static (RMT count, LEDC count, UART count) ใช้วิธี hardcoded limit ไม่ใช่ dynamic scoring 100 points

**แนวทางแก้ไข:**
- สร้าง scoring logic ตามตาราง handover_v1.12.0 ข้อ 7.2
- คำนวณคะแนนรวมทุกครั้งที่มีการแก้ไข outputs
- ถ้าเกิน 100 คะแนน → UI แสดง warning + block Save
- เพิ่ม endpoint `GET /api/scoring` คืนคะแนนปัจจุบันและ remaining
- แสดงคะแนนใน Web UI ข้างตาราง outputs

**ไฟล์ที่ต้องแก้ไข:**
- `include/output_control.h` — เพิ่มฟังก์ชัน `calculateScore(const OutputChannel& ch)` หรือ method
- `src/main.cpp` — เพิ่ม `/api/scoring`, ผูก scoring กับ `/api/outputs` POST validation
- `include/web_pages.h` — แสดงคะแนนใน UI

### 3.10 AC Dimmer-Backed 7-Segment

**อ้างอิง:** handover_v1.12.0 ข้อ 8.4 (Special Physical Segment Architectures)

**ปัญหาปัจจุบัน:** ไม่มีไดรเวอร์หรือสถาปัตยกรรมสำหรับขับ 7-segment ผ่าน AC Dimmer (Triac) channels

**แนวทางแก้ไข:**
- สร้าง output mode ใหม่ (type 14) หรือ extension ของ type 3 (AC Dimmer)
- ใช้ 7 channels (A-G) หรือ 8 channels (A-G + DP) ของ AC dimmer
- DMX 1 channel สำหรับค่าของ segment (0-9, A-Z ตาม font table)
- ต้อง map segment byte ไปเป็น 7-8 AC dimmer channels พร้อมกัน
- แนวทางนี้ต้องต่อ AC dimmer board แบบ 7-8 channels

**ไฟล์ที่ต้องแก้ไข:**
- `include/dimmer_control.h` — เพิ่มโหมด 7-segment display
- `include/output_control.h` — เพิ่ม type 14 หรือโหมดใหม่ของ AC dimmer
- `include/motion_control.h` — เพิ่ม driver
- `include/web_pages.h` — เพิ่ม UI

## 17. I2C Mutex & Scan Fix (2026-06-17)

### 17.1 Bus Contention (I2C Mutex)
- **ปัญหา:** `Wire` ไม่มี mutex — I2C scan ทำงานใน async HTTP handler (core ใดก็ได้) ขณะที่ PCA9685/expander writes ทำงานจาก main loop พร้อมกัน → bus สัญญาณรบกวน → scan เจออุปกรณ์เกินจริงหรือไม่เจอ
- **แก้:** เพิ่ม `SemaphoreHandle_t i2cMutex` ครอบ `Wire.beginTransmission()..endTransmission()` ทุกจุด
  - `config.h` — `#include <freertos/FreeRTOS.h>`, `#include <freertos/semphr.h>`, `extern SemaphoreHandle_t i2cMutex;`
  - `main.cpp` — global `SemaphoreHandle_t i2cMutex = NULL;`, สร้างใน `setup()` หลัง `Wire.begin()`, ครอบ I2C scan loop ทั้งหมด
  - `i2c_gpio_expander.h` — ครอบ writeReg8/16, writePcf, regRead8/16, readInputs (PCF)

## 23. บทวิเคราะห์เชิงลึก: สาเหตุที่คอนโทรลเลอร์ค้าง (Stuck Status LED / Freeze)

จากการวิเคราะห์ระบบการทำงานและปัญหาคอนโทรลเลอร์ค้าง (ไฟสถานะติดค้าง/เชื่อมต่อเครือข่ายไม่ได้) มีสาเหตุหลักที่เป็นไปได้ 4 ข้อดังนี้:

### 23.1 GPIO 12 (MTDI) Strapping Pin Conflict
- **สาเหตุ:** ขา GPIO 12 ของ ESP32 เป็นขา **Strapping Pin** ซึ่งใช้ตรวจสถานะแรงดันไฟฟ้าสำหรับชิป Flash SPI (Vdd_SDIO) ในตอนเริ่มต้นบูต
- **กลไกการค้าง:** หากขานี้ถูกดึงกระแสเป็น **HIGH** (จากวงจรภายนอก เช่น อุปกรณ์ขับมอเตอร์/หรี่ไฟ หรือโมดูล segment ที่เชื่อมต่ออยู่) ในตอนเปิดเครื่องหรือรีเซ็ต ESP32 จะเข้าใจว่าต้องจ่ายไฟให้ Flash Memory ที่แรงดัน **1.8V** แทนที่จะเป็น **3.3V** ส่งผลให้ชิปไมโครคอนโทรลเลอร์เกิด **Flash read error** ทันทีในขั้นตอน Bootloader ทำให้คอนโทรลเลอร์ค้างตั้งแต่เริ่มแรกและไฟสถานะค้าง (เนื่องจากโค้ดของแอปพลิเคชันยังไม่ทันได้เริ่มรันและไม่สามารถเพิ่มจำนวน Boot count เพื่อเข้า Recovery Mode ได้)
- **แนวทางแก้ไข:** หลีกเลี่ยงการใช้ GPIO 12 เป็นขาเอาต์พุตสำหรับควบคุมอุปกรณ์ที่มีตัวต้านทาน Pull-up ภายนอก หรือเปลี่ยนไปใช้งานขาที่ปลอดภัยกว่า เช่น GPIO 2, 32, 33

### 23.2 I2C SDA/SCL Conflict (GPIO 14 & 15)
- **สาเหตุ:** บอร์ดใช้ GPIO 14 เป็น I2C SDA และ GPIO 15 เป็น I2C SCL สำหรับสื่อสารกับอุปกรณ์ขยายช่องสัญญาณ (เช่น PCA9685, MCP23017)
- **กลไกการค้าง:** หากผู้ใช้งานตั้งค่าช่องเอาต์พุตประเภท DMX, Relay หรือ Dimmer มาใช้งานขา GPIO 14 หรือ 15 โดยตรง จะทำให้พินถูกดึงสถานะไปเป็นเอาต์พุตอิสระ เมื่อระบบเริ่มการทำงานของบัส I2C จะทำให้สายสัญญาณติดขัดและค้างที่ฟังก์ชันส่งข้อมูล I2C (Bus Busy Deadlock) ส่งผลให้ตัวควบคุมหลักหยุดทำงานและค้างไปทันที
- **แนวทางแก้ไข:** ระบบได้มีระบบ Validation เพื่อแจ้งเตือนบล็อกไม่ให้เพิ่มแชนเนลทับขา I2C (หากตั้งค่าใช้งาน I2C อยู่) แต่ผู้พัฒนาต้องระวังไม่เขียนข้ามผ่านการตรวจสอบ API โดยตรง

### 23.3 Zero-Crossing Interrupt Storm
- **สาเหตุ:** ช่อง AC Dimmer จำเป็นต้องอ่านสัญญาณ Zero-Crossing จากขารับสัญญาณอินพุต (ZC Pin)
- **กลไกการค้าง:** หากกำหนด ZC Pin ผิดพลาด หรือสายรับสัญญาณไม่มีการเชื่อมต่อจริง (ปล่อยลอย Floating) สัญญาณรบกวนความถี่สูงอาจทำให้เกิด Hardware Interrupt รัวๆ ไปที่ CPU (Interrupt Storm) จนทำให้ CPU ไม่มีรอบเวลาสำหรับทำงานใน Thread หลัก (รวมถึงส่วนเชื่อมต่อเครือข่ายและลูปควบคุม) ส่งผลให้เกิดการกระตุ้นซอฟต์แวร์รีเซ็ตจาก Watchdog หรือค้างไปดื้อๆ
- **แนวทางแก้ไข:** ตรวจสอบคอนฟิก ZC Pin ให้ถูกต้อง หรือตั้งค่าเป็น 255 (Disable) หากไม่ได้เชื่อมต่อบอร์ดหรี่ไฟจริง

### 23.4 Brownout Reset จากวิทยุ Wi-Fi
- **สาเหตุ:** บอร์ด WT32-ETH01 กินกระแสสูงมากในขณะที่ระบบวิทยุ (Wi-Fi/AP) เริ่มทำงาน
- **กลไกการค้าง:** แหล่งจ่ายไฟที่เลี้ยงบอร์ดหากจ่ายกระแสได้ไม่เพียงพอ จะทำให้แรงดันตกลงไปต่ำกว่าเกณฑ์ความปลอดภัยของ ESP32 (Brownout threshold) เกิดการวนลูปรีเซ็ตตัวเองไม่สิ้นสุด
- **แนวทางแก้ไข:** การรันในโหมด Recovery Mode จึงเลือกปิด Wi-Fi/AP และบังคับให้รันผ่านระบบสายแลน Ethernet เพียงอย่างเดียวเพื่อป้องกันปัญหากระแสตกคร่อมนี้
