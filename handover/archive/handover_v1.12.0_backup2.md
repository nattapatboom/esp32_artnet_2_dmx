## 3. แผนการใช้งานพอร์ตขยาย I2C (I2C Expander Architecture Plan)

เพื่อเพิ่มทรัพยากรพินเอาต์พุตให้เพียงพอกับ 16 ช่องใช้งาน และเพื่อประหยัดจำนวน GPIO บน ESP32 จึงมีแผนโครงสร้างของระบบ Virtual Pin ขยายตามแนวทางนี้:

### 3.1 ตัวเลือกชิปขยาย I2C ยอดนิยมในตลาด (Popular Expander ICs)
*   **PCA9685 (16-Ch PWM, 12-bit):** ชิปขยายช่องสัญญาณ PWM ความถี่สูงอิสระ มีความละเอียด 12-bit เหมาะที่สุดสำหรับการขับ RC Servo และไฟดีมเมอร์ PWM หรี่หลอดไฟ/LED (ผ่านมอสเฟต)
*   **MCP23017 / TCA9555 (16-Ch Digital I/O):** ชิปขยายดิจิทัลพิน 16 ขา มีความเสถียรสูง มีขา Interrupt แยก เหมาะกับงานสั่งเปิดรีเลย์ โซลินอยด์ หรือคุมขาทิศทางมอเตอร์
*   **PCF8575 / PCF8574 (16/8-Ch Low-Cost I/O):** ชิปขยายราคาย่อมเยา ขาทำงานแบบ Quasi-bidirectional เหมาะกับงานเปิด/ปิดรีเลย์บอร์ดขนาดเล็ก
*   **MCP23008 (8-Ch Digital I/O):** ชิปขยายดิจิทัลพิน 8 ขา สำหรับระบบที่ต้องการความเสถียรของสถานะพินและมีขนาดกะทัดรัด

### 3.3 สถาปัตยกรรมแบบผสม (Mixed/Hybrid Output Mapping)

เพื่อประสิทธิภาพในการสร้างสัญญาณความถี่สูงและประหยัดขาสัญญาณของ ESP32 แนะนำให้ใช้โครงสร้างแบบ **Hybrid Pin Mapping** ดังนี้:

#### 1. โหมดควบคุมสเต็ปเปอร์ (Stepper Motor Hybrid Mapping)
*   **ขา STEP (Pulse):** **ต้องต่อเข้ากับ GPIO ของ ESP32 โดยตรงเท่านั้น**
    *   *เหตุผล:* การสลับสัญญาณ STEP (High/Low) ทำงานด้วยความถี่สูง (1kHz - 20kHz+) และมีจังหวะเวลา (timing) ที่ต้องนิ่งและรวดเร็ว หากสั่งงานผ่านบัส I2C จะเกิด Bus Latency (ประมาณ 0.5-2ms) ทำให้มอเตอร์หมุนกระตุก ไม่เรียบ หรือเกิดอาการสูญเสียจังหวะ (Step Loss)
*   **ขา DIR (Direction) และ ขา ENABLE:** **สามารถต่อผ่าน I2C Expander ได้ทุกตัว**
    *   *เหตุผล:* สัญญาณทิศทางและเปิด/ปิดบอร์ดไดรเวอร์ เปลี่ยนสถานะค่อนข้างช้า ทำเฉพาะตอนเริ่มต้นขยับมอเตอร์หรือเปลี่ยนทิศทางเท่านั้น จึงไม่มีปัญหาเรื่อง Bus Latency
    *   *ตัวอย่างการจัดพิน (Wiring Example for 2 Steppers):*
        *   **Stepper 1:** STEP $\rightarrow$ `ESP32 GPIO 4` | DIR $\rightarrow$ `MCP23017 Port A0` | ENABLE $\rightarrow$ `MCP23017 Port A1`
        *   **Stepper 2:** STEP $\rightarrow$ `ESP32 GPIO 5` | DIR $\rightarrow$ `MCP23017 Port A2` | ENABLE $\rightarrow$ `MCP23017 Port A3`
        *   *ผลลัพธ์:* ประหยัดพินหลักของ ESP32 ได้ถึง 4 พินต่อมอเตอร์ 2 ตัว!

#### 2. สวิตช์ลิมิตชนร่วมกับโฮมมิ่ง (Homing Limit Switches & Interrupts)
*   เพื่อป้องกันมอเตอร์วิ่งเลยขอบเขตและระบุตำแหน่งเริ่มต้น (Home position)
*   หากต่อสวิตช์ลิมิตเข้ากับ GPIO Expander (เช่น MCP23017) แนะนำให้ต่อพิน **INTA** หรือ **INTB** (Interrupt Output) ของ MCP23017 กลับมายังขา GPIO ของ ESP32 จำนวน 1 ขา (เช่น GPIO 12)
*   *การทำงาน:* เมื่อลิมิตสวิตช์ถูกกด MCP23017 จะดึงขา Interrupt ต่ำลงทันที ESP32 จะจับสัญญาณอินเทอร์รัปต์นี้แล้วค่อยส่งคำสั่งอ่านค่าพอร์ตผ่านบัส I2C เพื่อระบุว่าสวิตช์ตัวไหนชน วิธีนี้ทำให้การตอบสนองรวดเร็วกว่าการทำ Polling วนลูปอ่านค่าบัส I2C ตลอดเวลา และช่วยลดทราฟฟิกบนบัส I2C

## 4. สถาปัตยกรรมและการรองรับไฟ LED Strip (LED Strip Architecture)

เพื่อตอบสนองการใช้งานร่วมกับไฟ LED Strip หลายประเภทในตลาด ในแผนออกแบบมีการกำหนดโครงสร้างการรองรับดังนี้:

### 4.1 การจำแนกตามโปรโตคอลความถี่และประเภทชิป
*   **กลุ่มความเร็วสูง 800kHz (Standard High-Speed):** ครอบคลุม **WS2812B, WS2813, WS2815, SK6812 RGB, GS1903, UCS1903** ขับข้อมูลโดยใช้ method มาตรฐานของ `NeoPixelBus` (`NeoEsp32RmtXWs2812xMethod` ทำงานผ่าน RMT ของ ESP32)
*   **กลุ่มความเร็วต่ำ 400kHz (Low-Speed):** ครอบคลุม **WS2811 รุ่นเก่า** (โดยเฉพาะแบบไฟ 12V/24V บางรุ่น) ซึ่งต้องการความกว้างของพัลส์ที่ยาวกว่า ในอนาคตควรเพิ่มตัวเลือกโปรโตคอล `LED_CHIP_TYPE` เพื่อให้ซอฟต์แวร์สลับไปใช้ `NeoEsp32RmtXWs2811Method` ได้
*   **กลุ่มไฟ RGBW (4 แชนเนล):** ครอบคลุม **SK6812 RGBW** ใช้โครงสร้างหน่วยความจำ 4 ไบต์ต่อพิกเซล และรองรับแอดเดรสสีขาวแยกต่างหาก

### 4.2 การสลับลำดับสีผ่าน Web UI (Color Order Configuration)
แต่ละรุ่นของ LED Strip มีการจัดวางพิกเซลสีภายในต่างกัน บอร์ดจึงรองรับการปรับค่าผ่านเว็บดังนี้:
*   `GRB`: ค่าเริ่มต้นสำหรับ WS2812B, WS2813, WS2815, SK6812 RGB
*   `RGB`: นิยมใช้ใน WS2811 บางประเภท
*   `BRG` หรือ `RBG`: สำหรับชิปสัญชาติจีน เช่น GS1903, UCS1903
*   `RGBW` / `GRBW` / `BRGW` / `WRGB`: สำหรับไฟแบบมีสีขาวผสม (SK6812 RGBW)

### 4.3 ข้อแนะนำในการติดตั้งสายสัญญาณและพลังงาน (Wiring & Hardware Level Shifter)
*   **ไฟเลี้ยง (Power):** WS2812B/SK6812 ใช้ไฟ 5V, WS2811 ใช้ไฟ 12V, WS2815 ใช้ไฟ 12V (มีสาย Backup data) ต้องต่อกราวด์ (GND) ร่วมกับบอร์ด WT32-ETH01 เสมอ
*   **ระดับสัญญาณ Data (Logic Level Shifting):** ชิป LED Strip ส่วนใหญ่ต้องการลอจิก Data ที่ 5V แต่ ESP32 ปล่อยสัญญาณที่ 3.3V แม้การต่อตรงอาจทำงานได้ในระยะสั้น แต่เพื่อความเสถียรในงานติดตั้งจริง **จำเป็นต้องใช้ไอซี Level Shifter (เช่น 74AHCT125 หรือ TXS0108E)** แปลงสัญญาณ 3.3V เป็น 5V ก่อนเข้าสาย Data ของไฟเส้น เพื่อป้องกันปัญหาไฟกระพริบเมื่อเดินสายสัญญาณยาวเกิน 1-2 เมตร
*   **ขารองรับการสำรองข้อมูล (Backup Data Pin):** ไฟอย่าง WS2813 และ WS2815 จะมีสายสัญญาณ 4 เส้น (V+, GND, DI, BI) โดยขา BI (Backup Input) ให้ต่อลงกราวด์ (GND) ของจุดเริ่มต้นไฟ เพื่อให้สัญญาณ DI ทำงานเป็นหลัก

## 5. ส่วนเพิ่มเติม: แผนการพัฒนาหน้าจอแสดงผลบนบัส I2C (I2C Display Implementation & Diagnostics Plan)

เนื่องจากหน้าจอ I2C เป็นฟีเจอร์สำคัญสำหรับแสดงสถานะเน็ตเวิร์กและ Art-Net โดยตรงหน้าเครื่อง ในแผนออกแบบ v1.12.0 จึงขอกำหนดแนวทางการทำงานร่วมกันบน I2C ดังนี้:

### 5.1 ประเภทฮาร์ดแวร์หน้าจอที่ออกแบบรองรับ
*   **OLED Display (SSD1306 / SH1106):** ขนาด 0.96 นิ้ว หรือ 1.3 นิ้ว ความละเอียด 128x64 พิกเซล เชื่อมต่อผ่านบัส I2C แอดเดรสเริ่มต้น `0x3C` หรือ `0x3D` (เหมาะสำหรับแสดงกราฟิกสถานะ ไอคอน และข้อมูล IP Address ที่ชัดเจน)
*   **Character LCD 16x2 / 20x4:** ทำงานร่วมกับชิปขยายขา I2C Backpack (PCF8574) แอดเดรสเริ่มต้น `0x27` หรือ `0x3F` (เหมาะสำหรับงานตู้คอนโทรลที่ต้องการความทนทาน ตัวอักษรขนาดใหญ่ อ่านง่ายในระยะไกล)

### 5.2 การใช้งาน I2C บัสร่วมกัน (Shared Bus Optimization)
บอร์ด WT32-ETH01 กำหนดให้หน้าจอแสดงผลใช้สัญญาณร่วมบนบัส I2C พอร์ตหลักพอร์ตเดียวกัน (SDA = GPIO14, SCL = GPIO15) ร่วมกับบอร์ดขยายพอร์ตตัวอื่นๆ (เช่น PCA9685, MCP23017) เพื่อเป็นการประหยัดพิน โดยมีหลักการจัดลำดับการรับส่งข้อมูล (Traffic Management) เพื่อไม่ให้ขัดขวางจังหวะการเปิด/ปิดไฟเอาต์พุตดังนี้:

1.  **การควบคุม Bus Traffic & Latency:**
    *   *ปัญหา:* การวาดภาพส่งข้อมูลไปยังจอ OLED 128x64 จะต้องส่งข้อมูล Buffer ขนาด 1024 ไบต์ผ่านบัส I2C ทุกครั้ง ซึ่งทำให้บัสถูกครอบครองและเกิดการหน่วงเวลาการทำงานของบอร์ดขยายพอร์ตเอาต์พุตตัวอื่นๆ (Bus Latency สูงขึ้น)
    *   *แนวทางแก้ไข:* 
        *   กำหนดให้มีความถี่ในการรีเฟรชหน้าจอ (Screen Refresh Rate) อยู่ที่ **2 Hz ถึง 5 Hz** เท่านั้น (อัปเดตทุก 200ms - 500ms) หรืออัปเดตหน้าจอเฉพาะเมื่อระบบตรวจพบว่าข้อมูลสำคัญมีการเปลี่ยนแปลง (เช่น เมื่อได้ IP ใหม่ หรือเมื่อ Art-Net หยุดรับส่งสัญญาณ)
        *   ตั้งความเร็วสัญญาณ I2C Bus ไว้ที่ **400 kHz** เป็นมาตรฐาน เพื่อย่นระยะเวลาการส่งเฟรมหน้าจอให้เร็วขึ้น
2.  **การหลีกเลี่ยงไลบรารีแบบ Blocking:**
    *   เลือกใช้ไลบรารีน้ำหนักเบาและทำงานแบบ Non-blocking เช่น `SSD1306Ascii` (สำหรับการแสดงผลเฉพาะตัวอักษรบน OLED เพื่อหลีกเลี่ยงการส่ง full-frame buffer) หรือใช้ไลบรารีที่สามารถทำ I2C transfer แบบเบื้องหลังเพื่อไม่ให้รบกวน Core 1 ซึ่งทำหน้าที่ขับเฟรมเอาต์พุต
3.  **ข้อมูลสถานะสำคัญที่จะแสดงผลบนหน้าจอ:**
    *   หน้าจอจะถูกแบ่งพื้นที่แสดงผลออกเป็น 4 ส่วนหลัก:
        *   **บรรทัดที่ 1:** IP Address ของตัวเครื่อง (Ethernet / WiFi SoftAP)
        *   **บรรทัดที่ 2:** สถานะการเชื่อมต่อเครือข่าย (Link Up / Down / DHCP Pending)
        *   **บรรทัดที่ 3:** Universe เริ่มต้น และ จำนวน Art-Net Universe ที่ตรวจจับได้จริง
        *   **บรรทัดที่ 4:** อัตราการแสดงผลเฟรมเรตเอาต์พุตปัจจุบัน (Output FPS) / อุณหภูมิสถานะชิปภายใน

## 6. ส่วนเพิ่มเติม: การปรับปรุง Web UI และระบบอัตราทดมอเตอร์ (Web UI & Motor Scale Enhancements)

เพื่อแก้ไขประเด็นความปลอดภัยในการป้อนค่าหน้าเว็บและขีดความสามารถการใช้งานร่วมกับ AC Servo ที่ใช้จำนวนพัลส์ปริมาณสูง จึงขอกำหนดแนวทางการปรับปรุงต่อไปนี้ในแผนออกแบบ:

### 6.1 การปรับปรุงส่วนตั้งค่าความเร็วของระบบ (Web UI Config Improvements)
*   **ปุ่มเลือกความเร็ว I2C Speed (Dropdown Menu):**
    *   *แนวทาง:* ปรับเปลี่ยนช่องป้อนข้อมูล `I2C Speed` บนหน้าเว็บ จากเดิมที่เป็นกล่องข้อความกรอกตัวเลข (Text Input) ให้เปลี่ยนเป็น **Dropdown Select Menu** เพื่อลดความผิดพลาดในการกรอกค่าของยูสเซอร์ โดยมีตัวเลือกมาตรฐานดังนี้:
        *   `100 kHz` (Standard Mode - สำหรับสายยาวหรือระบบที่เน้นเสถียรภาพ)
        *   `400 kHz` (Fast Mode - แนะนำเป็นค่าเริ่มต้นสำหรับ Expander และ Display ร่วมกัน)
        *   `1000 kHz` (Fast Mode Plus - สำหรับการทดสอบสายส่งสัญญาณสั้นพิเศษและต้องการการตอบสนองระดับไมโครวินาที)

### 6.2 การปรับปรุงหน้าต่างทดสอบเอาต์พุต (Output Test Menu Fixes)
*   *ปัญหา:* ในส่วนการทดสอบโหมด Stepper/AC Servo ปัจจุบันเมื่อตั้งค่าแชนเนลในโหมด 32-bit Position (4 ไบต์) ช่องสำหรับทดสอบการทำงานบน Web UI ยังคงใช้ค่าลิมิตสูงสุดแค่ `255` (8-bit) ซึ่งไม่เพียงพอและทำให้ไม่สามารถทดสอบสั่งวิ่งไปยังพิกัดเป้าหมายจริงที่มีจำนวนพัลส์หลายหมื่นสเตปได้
*   *แนวทางแก้ไข:*
    *   ปรับปรุง JavaScript ในโมดูล Test Tool บน Web UI ให้ตรวจเช็กประเภทของเอาต์พุต
    *   หากตรวจพบว่าเป็นโหมด Stepper/AC Servo (32-bit Position) ให้สลับช่องป้อนค่าทดสอบจาก Slider 0-255 ไปเป็น **Numeric Input Box ที่รองรับการคีย์ค่าตำแหน่งพิกัดแบบลงลึก 32-bit (รองรับตัวเลขเป้าหมายได้ตั้งแต่ -2,147,483,648 ถึง 2,147,483,647)** เพื่อส่งแพ็กเก็ตทดสอบโครงสร้าง 4 ไบต์ไปยังบอร์ดตรงตัวผ่าน API `/api/output-test`

### 6.3 ระบบคำนวณอัตราทดและการแปลงหน่วย (Software Scale Factor & Unit Conversion)
เนื่องจากการต่อใช้งานมอเตอร์ Stepper ร่วมกับเกียร์ทด หรือการนำไปใช้ควบคุม AC Servo ที่มักจะมี Resolution สูงมาก (เช่น ตั้งค่าพัลส์รับสัญญาณไว้ที่ 10,000 pulses/rev) ทำให้ผู้ใช้ควบคุมตำแหน่งได้ยากหากต้องคำนวณหาจำนวนพัลส์ด้วยตนเอง จึงกำหนดสถาปัตยกรรมการแปลงค่าดังนี้:

1.  **อัตราการคูณเชิงกายภาพ (Scale Factor):**
    *   เพิ่มช่องกรอกข้อมูลทศนิยม `Scale Factor` ในการตั้งค่ามอเตอร์บนหน้าเว็บ (เช่น `Scale Factor = 80.0` หรือ `Scale Factor = 0.036`)
    *   *สูตรคำนวณในเฟิร์มแวร์:*
        $$\text{Target Steps} = \text{DMX Position Value} \times \text{Scale Factor}$$
2.  **การแปลงและแสดงผลหน่วยปลายทางบน Web UI (Unit Translation):**
    *   เพิ่ม Dropdown ตัวเลือก **Unit Type** ในระบบตั้งค่ามอเตอร์:
        *   `Steps` (จำนวนพัลส์พิกเซลดิบ)
        *   `Degrees (°)` (องศาการหมุน - เช่น สำหรับงาน Pan/Tilt)
        *   `Millimeters (mm)` (มิลลิเมตร - สำหรับการขับแท่นสไลด์เชิงเส้น Linear Guide)
    *   *การตั้งค่าตัวอย่าง:*
        *   **องศาการหมุน (Degrees):** หากมอเตอร์ขับตรงและมีสเตป 200 steps/rev (1.8° ต่อสเตป) ผู้ใช้ต้องการสั่งงาน DMX เป็นค่าองศา (0 ถึง 360) ตัวคูณ Scale Factor จะถูกคิดเป็น $\frac{200}{360} \approx 0.555$ เมื่อสั่ง DMX = 180 (องศา) บอร์ดจะหมุนจ่ายไฟ 100 พัลส์เพื่อทำมุม 180° ทันที
        *   **ระยะเชิงเส้น (mm):** หากแท่นสไลด์ขับด้วยสายพานที่มีระยะ Pitch 40 mm ต่อการหมุน 1 รอบ (10,000 pulses/rev สำหรับ AC Servo) ผู้ใช้ต้องการระบุพิกัดเป็น mm (0-500 mm) ตัวคูณ Scale Factor จะคิดเป็น $\frac{10000}{40} = 250$ เมื่อสั่ง DMX = 120 (mm) บอร์ดจะสั่งหมุน 30,000 พัลส์เพื่อเลื่อนแท่นสไลด์ไปที่ตำแหน่ง 120 mm
    *   เมื่อเปิดใช้งาน ตัวช่วยทดสอบ (Test Tool) และการตรวจสอบสถานะตำแหน่งจริง (Telemetry) บน Web UI จะแสดงหน่วยกำกับปลายทางเป็น `mm` หรือ `°` อัตโนมัติ เพื่อสร้างความเข้าใจที่ง่ายและป้องกันความสับสน

## 7. ส่วนเพิ่มเติม: ระบบจัดสรรโควตาเอาต์พุตแบบไดนามิก (Dynamic Output Scoring System Plan)

เพื่อป้องกันปัญหาทรัพยากรของบอร์ด (RAM / CPU / Hardware Timers) ล้นจนระบบแครช หรือทำให้เฟรมเอาต์พุตกระตุก ในขณะเดียวกันก็เพิ่มความยืดหยุ่นในการเพิ่มจำนวนช่องใช้งานของอุปกรณ์ที่กินทรัพยากรต่ำ (เช่น รีเลย์) แผนการออกแบบ v1.12.0 ขอเสนอแนวทางการจัดสรรโควตาช่องเอาต์พุตแบบ **Dynamic scoring system (ระบบคะแนนสะสม)** ดังนี้:

### 7.1 หลักการจัดสรรคะแนนทรัพยากร (Physical Limit & Scoring Logic)
*   **แต้มทรัพยากรรวมของระบบ (Total System Resource Pool):** กำหนดให้บอร์ดมีแต้มทรัพยากรรวมสูงสุด = **100 คะแนน (Resource Points)**
*   ผู้ใช้สามารถสร้างสล็อตเอาต์พุต (สูงสุด 16-32 ช่องตามขีดจำกัดหน่วยความจำจัดเก็บในไฟล์คอนฟิก) ได้อย่างอิสระ แต่คะแนนรวมของเอาต์พุตทั้งหมดในระบบ **ห้ามเกิน 100 คะแนน** หากเกินระบบจะแจ้งเตือนและบล็อกไม่ให้ทำการบันทึกค่า

### 7.2 เกณฑ์การคำนวณคะแนนตามภาระงานจริง (Resource Scoring Matrix)

| ประเภทเอาต์พุต (Output Type) | คะแนนฐาน (Base Points) | คะแนนแปรผันตามขนาด (Variable Points) | เหตุผลและผลกระทบต่อทรัพยากร (Resource Justification) |
| :--- | :---: | :--- | :--- |
| **Relay / Solenoid (On/Off)** | **0.5** | ไม่มี | ใช้ทรัพยากรต่ำมาก เป็นการดึงสถานะลอจิกพินทั่วไป ไม่ใช้ Timer/Interrupt และกิน RAM น้อยกว่า 10 ไบต์ต่อช่อง |
| **LED Strip (RGB / RGBW)** | **2.0** | **+1.0 คะแนน** ต่อทุกๆ 100 พิกเซล | ใช้ RAM สูง (1 พิกเซลกิน RAM 6 ไบต์บนบอร์ด) และต้องแบ่งใช้ทรัพยากร RMT Hardware Block ของ ESP32 (จำกัด RMT สูงสุด 8 ช่อง) |
| **Stepper Motor (STEP/DIR)** | **8.0** | ไม่มี | ใช้ CPU สูงมากในการคำนวณและสร้างพัลส์ความเร่งแบบเรียลไทม์ และกินช่องสัญญาณ Interrupt ของ Hardware Timer |
| **DMX512 Output (Serial)** | **4.0** | ไม่มี | กินพื้นที่บัฟเฟอร์ DMX 512 ไบต์ และใช้ UART hardware / RMT block ในการรักษาความเร็ว 250kbps |
| **DC Motor (PWM/LEDC)** | **3.0** | ไม่มี | ใช้ช่องสัญญาณสร้างความถี่ (LEDC hardware channel) ซึ่ง ESP32 มีจำกัดที่ 16 ช่อง |
| **RC Servo Motor (LEDC)** | **2.0** | ไม่มี | ใช้ช่องสัญญาณสร้างความถี่ (LEDC hardware channel) ความถี่คงที่ 50Hz |
| **TM1637 Display / Buzzer** | **1.5** | ไม่มี | มีการใช้ฟังก์ชันเวลาและคำนวณเสียงคีย์ตัวโน้ตเป็นระยะ |

### 7.3 ตัวอย่างการคำนวณการจัดโหลดของระบบ (Configuration Examples)
*   **กรณีที่ 1: ระบบรีเลย์ขนาดใหญ่**
    *   ผู้ใช้ต้องการต่อบอร์ดรีเลย์ 32 ช่อง $\rightarrow$ $32 \times 0.5 = 16$ คะแนน (ผ่านบัสขยายพอร์ต I2C)
    *   *ผลลัพธ์:* ใช้ไปเพียง 16 คะแนน บอร์ดทำงานได้อย่างเสถียร สบายๆ เหลือแต้มอีกมาก
*   **กรณีที่ 2: ระบบขับไฟเส้น LED Strip หนาแน่นสูง**
    *   ผู้ใช้ต้องการต่อไฟเส้น 4 ช่อง ช่องละ 200 พิกเซล $\rightarrow$ ช่องละ $(2 + 2) = 4$ คะแนน $\rightarrow$ รวม 16 คะแนน
    *   บอร์ด DMX เอาต์พุต 2 ช่อง $\rightarrow$ $2 \times 4 = 8$ คะแนน
    *   มอเตอร์ Stepper คอนโทรล 2 แกน $\rightarrow$ $2 \times 8 = 16$ คะแนน
    *   *คะแนนรวม:* $16 + 8 + 16 = 40$ คะแนน
    *   *ผลลัพธ์:* ผ่านเกณฑ์สบายๆ (ระบบทำงานได้ลื่นไหล ไม่กระตุก)
*   **กรณีที่ 3: เกินขีดจำกัด (Overloaded - บล็อกการบันทึก)**
    *   ผู้ใช้พยายามตั้งค่าไฟเส้นขนาดใหญ่ 5 ช่อง ช่องละ 1000 พิกเซล $\rightarrow$ ช่องละ $(2 + 10) = 12$ คะแนน $\rightarrow$ รวม 60 คะแนน
    *   บอร์ด DMX เอาต์พุต 2 ช่อง $\rightarrow$ $2 \times 4 = 8$ คะแนน
    *   มอเตอร์ Stepper คอนโทรล 6 แกน $\rightarrow$ $6 \times 8 = 48$ คะแนน
    *   *คะแนนรวม:* $60 + 8 + 48 = 116$ คะแนน (เกิน 100 คะแนน)
    *   *ผลลัพธ์:* Web UI จะแสดงข้อความแจ้งเตือนสีแดง **"เกินขีดจำกัดทรัพยากรของระบบ (116/100 Points)"** และบล็อกปุ่ม Save เพื่อป้องกันไม่ให้ระบบล่มหรือแครชกลางคัน

### 7.4 หมายเหตุและการขยายโหมดอื่นๆ (Extensibility to Other Output Modes)
*   **ตารางคะแนนในหัวข้อ 7.2 เป็นเพียงตัวอย่างแนวทางการคิดคะแนนของโหมดหลักเท่านั้น**
*   ในการพัฒนาจริง ระบบคะแนนสะสมจะถูกพัฒนาให้ครอบคลุมทุกโหมดการทำงานของอุปกรณ์ (รวมถึงโหมด Sequential Smoke Shooter, โหมด Analog RGB/RGBW, โหมด Modbus TCP/RTU และโหมดอื่นๆ ในอนาคต) โดยจะทำการจัดสรรคะแนนให้เหมาะสมตามสัดส่วนการดึงหน่วยความจำและ CPU จริงของโปรโตคอลและโหมดการทำงานนั้นๆ

## 8. ส่วนเพิ่มเติม: การออกแบบและฟอนต์อักษรสำหรับหน้าจอ 7-Segment (7-Segment Pin Layout & Alphabet Font Design)

เพื่อเพิ่มความชัดเจนในการต่อสัญญาณ 7-Segment และความสามารถในการแสดงผลตัวอักษรเพื่อการแจ้งเตือนสถานะต่างๆ แผนการออกแบบ v1.12.0 มีการกำหนดโครงสร้างดังนี้:

### 8.1 สถาปัตยกรรมการต่อใช้งานพิน (7-Pin / 8-Pin Direct Drive Layout)
นอกจากโมดูลสำเร็จรูปประเภทสื่อสาร 2 พิน (TM1637) แล้ว ระบบจะรองรับการเชื่อมต่อเอาต์พุตขับหน้าจอ 7-Segment แบบต่อพินตรง (Direct Pin Drive) เพื่อความยืดหยุ่นในการสร้างวงจรควบคุมเอง:
*   **7-Pin Direct Drive (ไม่มีจุดทศนิยม):**
    *   ใช้ GPIO ของ ESP32 หรือพอร์ตดิจิทัลของ GPIO Expander จำนวน 7 พิน เพื่อทำหน้าที่ควบคุมแต่ละเซกเมนต์ (A, B, C, D, E, F, G) โดยตรง
    *   ระบบสนับสนุนการสลับชนิดของหน้าจอผ่าน Web UI:
        *   *Common Cathode (CC - ขั้วลบร่วม):* จ่ายลอจิก HIGH (`1`) เพื่อให้เซกเมนต์ติดไฟ
        *   *Common Anode (CA - ขั้วบวกร่วม):* จ่ายลอจิก LOW (`0`) เพื่อให้เซกเมนต์ติดไฟ
*   **8-Pin Direct Drive (รวมจุดทศนิยม DP):**
    *   เพิ่มพินขาสัญญาณที่ 8 เพื่อควบคุมจุดทศนิยม `DP` (Decimal Point) ทำให้หน้าจอแสดงจุดระบุทศนิยม หรือกระพริบเพื่อแสดงสถานะการรับส่งข้อมูลเน็ตเวิร์กได้
    *   *แผนผังการจับคู่พินมาตรฐาน (Default Segment to Pin Mapping):*
        *   Segment A $\rightarrow$ Pin 1
        *   Segment B $\rightarrow$ Pin 2
        *   Segment C $\rightarrow$ Pin 3
        *   Segment D $\rightarrow$ Pin 4
        *   Segment E $\rightarrow$ Pin 5
        *   Segment F $\rightarrow$ Pin 6
        *   Segment G $\rightarrow$ Pin 7
        *   Segment DP $\rightarrow$ Pin 8 (ใช้เฉพาะในโหมดขับแบบ 8 พิน)
    *   *คำแนะนำ:* การต่อตรงแบบ 7-8 พินจะใช้ GPIO ปริมาณมาก แนะนำให้เชื่อมต่อเอาต์พุตเหล่านี้ผ่าน **GPIO Expander (เช่น MCP23017, TCA9555 หรือ PCF8575)** เพื่อช่วยลดความต้องการใช้พินหลักของ ESP32 ได้ถึง 8 พินเต็ม

### 8.2 ฟอนต์การแสดงผลตัวอักษร A-Z และอักขระพิเศษ (7-Segment Font Table A-Z)
เพื่อใช้สำหรับแสดงข้อความแจ้งเตือนสถานะความผิดพลาด หรือรหัสสถานะในแบบ Diagnostics (เช่น "IP", "CONN", "ERR", "DMX", "TEST") บอร์ดจะบรรจุระบบแปลงรหัสไบนารีสำหรับแสดงผลตัวอักษรภาษาอังกฤษ A-Z (ตัวพิมพ์ใหญ่และตัวพิมพ์เล็กที่ดัดแปลงตามขีดจำกัดของ 7-Segment) ดังนี้:

*   **ตารางรหัสเซกเมนต์มาตรฐาน (Active High / Common Cathode):**
    *   การเรียงลำดับบิต: `[DP][G][F][E][D][C][B][A]` (บิต 0 = A (LSB), บิต 7 = DP (MSB))

| ตัวอักษร | รูปแบบเซกเมนต์ที่ติดสว่าง | รหัสไบนารี (DP -> A) | รหัสฐานสิบหก (Hex) |

| **F** | ติด A, E, F, G | `0b01110001` | `0x71` |
| **G** | ติด A, C, D, E, F, G | `0b00111101` | `0x3D` |
| **H** | ติด B, C, E, F, G (แสดงเป็น 'H' ตัวใหญ่) | `0b01110110` | `0x76` |

### 8.3 วัตถุประสงค์และขอบเขตการแสดงผล (Operational Scope & Art-Net Integration)
*   **ขอบเขตการแสดงผลบน 7-Segment:** 
    *   ฟอนต์ตัวอักษร A-Z และตัวเลขบนหน้าจอ 7-Segment (ทั้งผ่านไดรเวอร์ TM1637 และ Direct Drive 7-8 พิน) มีวัตถุประสงค์หลักเพื่อ **"รับรหัสข้อมูลตัวอักษร/คำสั่งจาก Art-Net (DMX) มาแสดงผลตามคำสั่งของซอฟต์แวร์ควบคุมภายนอกเท่านั้น"** (เช่น TouchDesigner หรือโปรแกรมคุมไฟสเตจส่งข้อความ DMX เข้ามาแสดงผลคำสั้นๆ)
*   **การแสดงสถานะการทำงานของบอร์ดเอง (System Diagnostics Display):**
    *   การแสดงผลค่าเครือข่ายของตัวเครื่อง (เช่น IP Address, สถานะ Ethernet Link, รหัสความผิดพลาดภายในบอร์ด) จะไม่มีการนำมาแสดงบน 7-Segment เพื่อป้องกันไม่ให้เกิดการสับสนหรือขัดจังหวะสัญญาณของผู้ใช้
    *   ระบบจะทำการส่งข้อมูลสถานะการทำงานของเครื่องเหล่านั้นไปแสดงผลเฉพาะบน **หน้าจอแสดงผลบัส I2C (OLED/LCD ในข้อ 5), หน้าแสดงผลของเว็บคอนโทรล (Web UI), และใช้ไฟกระพริบ LED Status บนบอร์ดที่มีอยู่แล้วเท่านั้น** ซึ่งเพียงพอต่อการตรวจสอบการทำงานทั่วไป

### 8.4 สถาปัตยกรรมทางกายภาพแบบพิเศษ (Special Physical Segment Architectures)
*   **7-Segment จากหลอดไฟ AC Dimmer (AC Dimmer-backed segments):**
    *   *แนวทางการขับงาน:* โครงสร้างของจอ 7-Segment ไม่จำเป็นต้องจำกัดอยู่เพียงโมดูล LED ขนาดเล็กหรือไอซีดิจิทัลเท่านั้น แต่อุปกรณ์สามารถขับสัญญาณเพื่อสร้าง 7-Segment ขนาดใหญ่ทางกายภาพ (เช่น ป้ายไฟยักษ์บนเวที) โดยการใช้ **"หลอดไฟกระแสสลับ (AC Light Bulbs) ที่ควบคุมความสว่างผ่านบอร์ดดีมเมอร์ AC Dimmer (Triac)"** จำนวน 7 หรือ 8 แชนเนล (Segment A-G + DP) ในการสลับความสว่างหรือหรี่ไฟเป็นตัวอักษรตามสัญญาณ DMX ที่ส่งมาจาก Art-Net ได้โดยตรง

### 8.5 แนวทางการเลือกรูปแบบฟอนต์และการอ่านตามบริบท (Segment Font Optimization & Context Readability)
*   **การเลือกรูปแบบตัวอักษรที่อ่านง่ายที่สุด (Readability Mapping):**
    *   เพื่อป้องกันความสับสนในการตีความและการซ้อนกันของลักษณะตัวเลข ระบบจะเลือกแสดงตัวอักษรพิมพ์ใหญ่หรือพิมพ์เล็กผสมผสานตามจุดที่สายตามนุษย์แยกแยะได้ง่ายที่สุด:
        *   `B` $\rightarrow$ แสดงเป็น **`b` ตัวเล็ก (`0x7C`)** เพื่อไม่ให้สับสนกับเลข `8` (`0x7F`)
        *   `D` $\rightarrow$ แสดงเป็น **`d` ตัวเล็ก (`0x5E`)** เพื่อไม่ให้สับสนกับเลข `0` (`0x3F`)
        *   `N` $\rightarrow$ แสดงเป็น **`n` ตัวเล็ก (`0x54`)** เพื่อแยกจากอักษร `H`
        *   `Q` $\rightarrow$ แสดงเป็น **`q` ตัวเล็ก (`0x67`)** เพื่อแยกจากอักษร `A`
        *   `R` $\rightarrow$ แสดงเป็น **`r` ตัวเล็ก (`0x50`)** และ `T` $\rightarrow$ แสดงเป็น **`t` ตัวเล็ก (`0x78`)** เพื่อความกระชับ
*   **การใช้ตัวอักษรใกล้เคียงสำหรับอักขระที่แสดงผลไม่ได้ (Fallback Characters for Complex Letters):**
    สำหรับอักขระบางตัวที่มีรูปทรงซับซ้อนเกินโครงสร้าง 7 ขีด ระบบจะใช้ตัวเลือกอักขระที่ใกล้เคียงที่สุด เพื่อให้ผู้ใช้อ่านและแปลความหมายจาก **"บริบทของข้อความ (Contextual Reading)"** ได้ง่าย:
    *   `K` $\rightarrow$ แสดงผลเลียนแบบด้วยอักษร **`H` (`0x76`)**
    *   `M` $\rightarrow$ แสดงผลเลียนแบบด้วยอักษร **`A` หรือ `n` (`0x37`)**
    *   `W` $\rightarrow$ แสดงผลเลียนแบบด้วยอักษร **`U` (`0x3E`)** หรือ `u` ตัวเล็ก
    *   `X` $\rightarrow$ แสดงผลเลียนแบบด้วยเครื่องหมายขีดกลาง **`[-]` (`0x40`)** (ตัวอย่าง: คำว่า `"TEXT"` จะแสดงผลเป็น `t` `E` `-` `t` ซึ่งเพียงพอให้อ่านเข้าใจความหมายจากบริบทแวดล้อมได้)

## 16. บันทึกเพิ่มเติม: PCA9685 Shared Frequency Policy

วันที่ดำเนินงาน: 2026-06-17

ตรวจสอบปัญหา frequency ร่วมของ PCA9685 แล้วพบว่าเป็นข้อจำกัดเฉพาะ PCA9685 เป็นหลัก เพราะทั้ง 16 channel ในชิปเดียวกันใช้ PWM frequency เดียวกัน ส่วน MCP23017/TCA9555/PCF8574/PCF8575 เป็น digital GPIO expander ไม่มี PWM frequency ร่วม จึงไม่มีปัญหาประเภทเดียวกันในโหมด on/off

### 16.1 Policy ที่ติดตั้งแล้ว
*   เพิ่ม logic ใน `OutputControl::getPcaSharedFrequency(address)`
*   หาก PCA9685 address เดียวกันมี RC Servo (type `8`) อย่างน้อย 1 output:
    *   ทั้ง PCA9685 chip/address นั้นจะถูกตั้งเป็น `50 Hz`
    *   Relay ที่อยู่บน chip เดียวกันใช้งานได้ปกติ เพราะเป็น full on/off
    *   DC Motor/PWM Dimmer/Analog LED บน chip เดียวกันจะถูกบังคับใช้ 50 Hz ด้วย ซึ่งอาจเกิดเสียงหึ่ง, torque ripple หรือ flicker ตามโหลด
*   หาก address นั้นไม่มี RC Servo:
    *   firmware ใช้ frequency ตัวแรกที่ตั้งไว้ในกลุ่ม PWM/Motor/Analog/Smoke ของ address นั้น
    *   ถ้ามีหลายความถี่ผสมกัน Web UI จะเตือนก่อนบันทึก เพราะ PCA9685 ใช้ได้แค่ frequency เดียวต่อ chip

### 16.2 Web UI Warning
*   ปรับ warning ใน `web_pages.h`
*   กรณีมี Servo บน PCA9685 address เดียวกับ PWM/Motor/Analog/Smoke ที่ไม่ใช่ 50 Hz:
    *   Web UI จะแจ้งว่า Servo บังคับทั้ง chip เป็น 50 Hz และถาม confirm ก่อนบันทึก
*   กรณีไม่มี Servo แต่มี mixed frequency:
    *   Web UI จะแจ้งว่า firmware จะใช้ shared frequency ตัวแรกของ address นั้น

## 17. Stepper Hybrid Output Mapping - Phase 1 Implemented

วันที่อัปเดต: 2026-06-17

อ้างอิงหัวข้อเดิม: `3.3 สถาปัตยกรรมแบบผสม (Mixed/Hybrid Output Mapping)`

สถานะ: ดำเนินการ Phase 1 แล้ว โดยเริ่มจาก Stepper Motor ก่อน เพื่อให้ขา STEP ยังอยู่บน ESP32 GPIO สำหรับ timing ที่แม่นยำ และให้ขา DIR/ENABLE สามารถย้ายไปอยู่บน I2C expander ได้

### 17.1 โครงสร้าง Config ที่เพิ่ม

เพิ่ม field สำหรับขาเสริมของ Stepper โดยยังคง backward-compatible กับ config เดิม:

- `pin2_source`, `pin2_addr`, `pin2_channel` สำหรับขา DIR
- `pin3_source`, `pin3_addr`, `pin3_channel` สำหรับขา ENABLE

ค่า `source` หลักของ Stepper ต้องเป็น `0` เท่านั้น เพราะหมายถึง STEP pin บน ESP32 GPIO

ค่า source ของขา DIR/ENABLE:

- `0` = ESP32 GPIO
- `1` = PCA9685
- `2` = MCP23017
- `3` = TCA9555
- `4` = PCF8574/PCF8575

### 17.2 Runtime Behavior

- `FastAccelStepper` ยังสร้าง pulse จาก `pin` หรือ STEP บน ESP32 GPIO เท่านั้น
- ถ้า DIR ใช้ ESP32 จะส่งให้ `FastAccelStepper::setDirectionPin()` เหมือนเดิม
- ถ้า DIR ใช้ expander เฟิร์มแวร์จะเขียน state ไปยัง expander ก่อนสั่ง `moveTo()`, `runForward()`, หรือ `runBackward()`
- ถ้า ENABLE ใช้ ESP32 จะใช้ `setEnablePin()` และ `setAutoEnable(true)` เหมือนเดิม
- ถ้า ENABLE ใช้ expander เฟิร์มแวร์จะเปิด enable ค้างไว้ขณะใช้งาน โดยตั้ง logic แบบ active-low ตาม driver stepper ทั่วไป

### 17.3 Web UI

หน้า Output Editor เพิ่มส่วน Stepper Hybrid Pins:

- Stepper DIR Source
- DIR Expander Address
- DIR Expander Channel
- Stepper ENABLE Source
- ENABLE Expander Address
- ENABLE Expander Channel

ตาราง Configured Channels แสดง Stepper แบบอ่านง่ายขึ้น เช่น:

`STEP GPIO 4 / DIR MCP23017 0x20:CH0 / EN MCP23017 0x20:CH1`

### 17.4 Validation

เพิ่ม validation ทั้งฝั่งหน้าเว็บและฝั่ง firmware API:

- Stepper `source` หลักต้องเป็น ESP32 GPIO เท่านั้น
- ถ้า DIR/ENABLE ใช้ expander ต้องเลือก channel ให้ครบ
- GPIO duplicate check จะไม่นับ DIR/ENABLE ที่ย้ายไป expander แล้ว
- Expander duplicate check จะนับ channel ของ DIR/ENABLE hybrid ด้วย เพื่อไม่ให้ชนกับ output อื่น
- Reserved GPIO เช่น Status LED, Zero-Crossing, I2C SDA/SCL จะยัง block เฉพาะขาที่เป็น ESP32 GPIO จริง

### 17.6 ข้อจำกัดที่ยังต้องทดสอบบน Hardware

### 18.1 Recommended Implementation Order

1. DC Motor Hybrid Mapping
   - PWM pin: ESP32 GPIO or PCA9685
   - DIR/IN pins: ESP32 GPIO or digital expander
   - Add per-pin source fields without breaking existing `source` config.

2. Stepper ENABLE polarity option
   - Add active-low / active-high setting.
   - Current expander ENABLE behavior assumes active-low.

3. Homing input expander support
   - Start with polling support for MCP23017/TCA9555/PCF857x.
   - Add optional interrupt pin later, especially for MCP23017.

4. Web UI cleanup
   - Group per-device pin mapping in each output mode.
   - Avoid showing PCA color/order fields or unrelated pin fields for non-matching modes.

### 18.2 Rule Summary for Future Code

- High-speed or strict timing signals must stay on ESP32: LED strip data, DMX TX, Stepper STEP, AC zero-cross, triac timing, buzzer PWM.
- PWM-like outputs can use ESP32 LEDC or PCA9685 only.
- Low-speed digital outputs can use MCP23017, TCA9555, or PCF857x.
- PCA9685 is not a generic digital expander; use it only when PWM/servo-style output is useful or when wasting a PWM channel is acceptable.
- PCA9685 frequency is shared per chip address; mixing servo and dimmer/motor on the same PCA chip forces 50 Hz.
- PCF857x is acceptable for simple outputs, but MCP23017/TCA9555 are cleaner choices for stable push-pull style output.
