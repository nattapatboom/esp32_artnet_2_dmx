## 3. Confirmed Bugs (ข้อบกพร่องที่ยืนยันแล้ว)

### 3.1 ZC Pin Interrupt Storm เมื่อตั้งค่า ≠ 255 แต่ไม่ต่อสาย
- **Severity:** **Critical** — ทำให้ controller freeze/crash
- **Location:** `include/dimmer_control.h:65`
- **Root Cause:** ESP32 hardware interrupt ถูก attach ไว้กับ pin ที่ floating → noise trigger interrupt รัวๆ → CPU ไม่มีเวลาให้ task อื่น → watchdog reset หรือ freeze
- **Current mitigation:** Code-level guard (return ทันทีถ้า zc_pin == 255), Web UI warning banner
- **Still missing:** ไม่มี hardware debounce หรือ interrupt rate-limiting ในกรณีที่สายหลุดขณะทำงาน
- **Suggestion:** ใช้ `gpio_set_intr_type()` + `gpio_intr_disable()` ร่วมกับ timer-based cooldown ถ้า interrupt มาเกิน X ครั้งใน 1 วินาที

### 3.2 GPIO 12 (MTDI) Strapping Pin Conflict
- **Severity:** **Critical** — ทำให้บอร์ดไม่บูตเลย
- **Location:** Hardware design (ไม่เกี่ยวกับ code)
- **Root Cause:** GPIO 12 เป็น strapping pin ถ้าถูกดึง HIGH ตอนบูต → ESP32 ตีความว่า Flash ต้องใช้ 1.8V → Flash read error → ค้าง
- **Current mitigation:** ไม่มีใน code — ต้องระวังตอน接线
- **Suggestion:** เพิ่ม warning ใน Web UI ตอนเลือก GPIO 12 สำหรับ output, หรือเพิ่ม startup check ใน `setup()` ว่า flash voltage register ถูกต้องหรือไม่

### 3.3 I2C Bus Deadlock เมื่อ SDA/SCL ถูกใช้เป็น GPIO output
- **Severity:** **High** — ทำให้ I2C ค้างทั้งระบบ
- **Location:** Hardware — ถ้า user ตั้ง output type ไปที่ GPIO 14 (SDA) หรือ GPIO 15 (SCL)
- **Root Cause:** เมื่อพิน I2C ถูก configure เป็น output โดย output driver → `Wire.beginTransmission()` จะรอ Bus clear ตลอดไป (blocking)
- **Current mitigation:** API validation บน main.cpp ตรวจและ block ถ้า pin ซ้ำกับ I2C pins
- **Still missing:** validation ไม่ครอบคลุมกรณี expander channel map ไปยัง I2C pins (expanders ใช้ I2C bus อยู่แล้ว ไม่ควรมีกรณีนี้ แต่ถ้า user ต่อ wrong wiring ก็เกิดได้)
- **Suggestion:** ตรวจสอบ I2C bus health ก่อน every I2C transaction (คล้ายกับ display probe pattern) ถ้า bus stuck → auto-disable I2C + report ทาง Web UI

### 3.4 LED Strip led_count Variable Score อาจคลาดเคลื่อน
- **Severity:** **Medium**
- **Location:** `include/scoring.h` — `scoreForType()` ใช้ `led_count/100` เป็น variable component
- **Root Cause:** ถ้า led_count = 0 → score = 2.0 (ไม่มี variable), ถ้า led_count ถูกตั้งเป็น default 170 → score = 3.7
- **Issue:** UI ไม่ validate ว่า `led_count` ต้อง > 0 หรือถูกจำกัด上限; UI warning สำหรับ > 1111 ต่อ strip และ > 4000 total แต่ scoring ไม่ได้ take into account 4000-pixel limit
- **Suggestion:** ทำให้ scoring และ resource warnings สอดคล้องกัน: ถ้า total pixels > 4000 → score penalty หรือ block save

## 4. Hardware Risks (ความเสี่ยงด้านฮาร์ดแวร์)

### 4.1 Stepper STEP Pin ต้องเป็น ESP32 GPIO โดยตรงเท่านั้น
- **Risk:** **High** — ถ้า user ต่อ STEP pin ผ่าน I2C expander → motor กระตุก/สูญเสียจังหวะ
- **Current:** Web UI มี expander address dropdown สำหรับ pin2/pin3 (DIR/EN) แต่ pin 1 (STEP) ยังคงเป็น ESP32 GPIO only — ถูกต้องแล้ว
- **Still missing:** ไม่มี warning ใน UI ถ้า user เลือก source ≠ 0 และ type = Stepper (type 7) → pin 1 ควรถูก force เป็น local badge
- **Reference:** v1.12.0 section 3.2

### 4.2 AC Dimmer และ LED Strip ต้องใช้ Level Shifter
- **Risk:** **Medium** — ไฟกระพริบ/สัญญาณไม่เสถียร ถ้าสาย Data ยาว > 1-2 เมตร
- **Current:** แค่คำแนะนำใน documentation (v1.12.0 section 4.3)
- **Suggestion:** เพิ่ม UI note "Recommend 74AHCT125 level shifter for long cable runs" ใน Output config สำหรับ type 0 (LED) และ type 3 (AC Dimmer)

### 4.3 Brownout จาก Wi-Fi + Ethernet พร้อมกัน
- **Risk:** **Medium** — ถ้า power supply จ่ายกระแสไม่พอ
- **Root Cause:** WT32-ETH01 มีทั้ง Wi-Fi และ Ethernet; ถ้าเปิด both + ต่อ expanders หลายตัว → peak current อาจเกิน PSU limit
- **Current:** v1.13 section 23.4 วิเคราะห์ไว้
- **Suggestion:** ใน Network tab แสดง estimated power draw หรืออย่างน้อย warning "Ensure power supply ≥ 2A for full configuration"

## 5. Code Quality / Technical Debt (หนี้ทางเทคนิค)

### 5.1 web_pages.h — Monolithic Embedded HTML+JS (~900+ lines)
- **Problem:** หน้า Web UI ทั้งหมด (Settings, Outputs, Diagnostics, Network, OTA) อยู่ใน `#define WEB_HTML` string เดียวใน C++ header
- **Consequences:**
  - แก้ไขยาก — ต้อง escape quotes, ไม่มี syntax highlighting ที่ดี
  - ไม่สามารถใช้ JS bundler/babel/ESLint ได้
  - ทุกการแก้ UI → rebuild firmware → OTA upload
- **Score:** **10/10 ควรแก้**
- **Suggestion:** ย้าย static assets ไป serve จาก LittleFS (SPIFFS) — build script สร้าง `.html.gz` → flash เป็น data partition
  แต่ก็ต้อง rebuild firmware ทุกครั้งเหมือนเดิม จริงๆ แล้วการเก็บใน PROGMEM เป็น pattern ปกติของ ESP32 embedded web
  **Practical fix:** แยก JS function blocks ออกเป็น sections ที่ indent และ comment ชัดเจนขึ้น

### 5.2 Config Storage — Dual System (Preferences + LittleFS)
- **Problem:** ระบบ settings ใช้ `Preferences` (NVS) สำหรับค่าเล็กน้อย (network, display config) และ `LittleFS` สำหรับ file-based outputs (`/outputs.json`)
- **Consequences:**
  - ต้อง maintain save/load 2 ระบบ
  - Factory Reset ต้องลบทั้ง NVS (Preferences.clear()) และ LittleFS
  - Preferences มีขนาดจำกัด (~4KB สำหรับ NVS partition)
- **Score:** **6/10 — acceptable但对于长期维护需要注意**
- **Suggestion:** ไม่ต้องเปลี่ยน — dual system เป็น practical สำหรับ embedded ถ้าทำงานดีอยู่

### 5.3 I2C Transaction — No Timeout on Bus Lock
- **Problem:** `Wire.beginTransmission()` / `Wire.endTransmission()` ไม่มี timeout — ถ้า bus lock จะ hang ตลอดไป
- **Current mitigation:** `i2cMutex` semaphore มี timeout (`portMAX_DELAY` = ตลอดไป) — ถ้า task ที่ถือ mutex ค้าง ทุก task จะรอตลอดไป
- **Score:** **7/10 ควรแก้**
- **Suggestion:** เปลี่ยนเป็น `xSemaphoreTake(i2cMutex, pdMS_TO_TICKS(100))` → ถ้าไม่ได้ mutex ใน 100ms → report error + skip transaction แทน blocking

### 5.4 output_control.h — Large Monolithic Header
- **Problem:** `include/output_control.h` รวม struct definition + save/load JSON + output type factory + `loop()` + `updateDimmer7Segs()` + `asciiToSegment()` — ทุกอย่างในไฟล์เดียว
- **Consequences:**
  - ทุกไฟล์ที่ include output_control.h จะ rebuild เมื่อมีการแก้ไขใดๆ
  - Namespace pollution
- **Score:** **5/10** — ควรแยกเป็น `output_control.h` (interface) + `output_control.cpp` (implementation) แต่ PlatformIO compile .cpp ใน src/ อยู่แล้ว
- **Suggestion:** ไม่เร่งด่วน — header-only pattern เหมาะกับ embedded (compiler inline optimization) มากกว่า

### 5.5 No Unit Tests / CI
- **Problem:** ทั้งโปรเจกต์ไม่มี automated test — ต้องพึ่ง hardware testing อย่างเดียว
- **Consequences:**
  - เปลี่ยน logic ใน scoring หรือ validation → ไม่แน่ใจว่าไม่ break อะไร
  - แต่ละรอบ dev → OTA upload → manual test
- **Score:** **8/10 ควรแก้ในระยะยาว**
- **Suggestion:** เริ่มจาก unit test สำหรับ scoring logic (`scoring.h` — pure functions, ไม่ depend on hardware) ด้วย Google Test บน host

## 7. Resource Constraints และ Future Scaling

### 7.1 Flash 65% — ยังพอเหลือสำหรับฟีเจอร์เล็กๆ
| Component | Flash |
|---|---|
| Framework + Libraries | ~50% (ฐาน) |
| Web UI (HTML+JS ใน PROGMEM) | ~8% |
| Output drivers + I2C + Scoring + Display | ~7% |
| **Remaining** | **~35% (687 KB)** |

### 7.2 RAM 17.5% — ยังสบาย แต่ต้องระวัง
- Main RAM consumer: NeoPixelBus buffers, DMX buffers, Ethernet stack
- ต่อให้เพิ่ม LED 4000 pixels → 4000×6bytes = 24KB → RAM ~25%
- **Risk:** `AsyncWebServer` + `Ethernet` + `WiFi` stacks กิน heap ใน `loop()` — ถ้ามี memory leak จะค่อยๆ ลดลง

### 7.3 RMT Channels — 8 Total
- LED Strip แต่ละเส้นกิน 1 RMT channel
- DMX (via RMT) → เพิ่มอีก
- ถ้า LED 7 เส้น + DMX 1 → 8/8 หมด
- **No mitigation ใน current code** (แค่ warning ใน resource check)

### 7.4 I2C Bus Load
- ถ้ามี PCA9685 (หลายตัว) + MCP23017 + Display + I2C Scan → bus load สูง
- โดยเฉพาะตอน display update (1024 bytes/frame) — ถ้า refresh 2Hz → ~2KB/s บน I2C
- **Current i2cMutex** ช่วยป้องกัน data corruption แต่ไม่ช่วยเรื่อง latency

## 10. Critical Code Paths ที่ต้องระวังในการแก้ไขครั้งต่อไป

1. **`POST /api/outputs`** — validation chain ยาว: parse JSON → validateOutputJson() → check expander conflict → check PCA9685 shared freq → check resource warnings → **check scoring** → save → reboot
   - การเพิ่ม validation step ต้องระวัง ordering และ error message ที่ชัดเจน

2. **`renderPinRows()` + `toggleOutFields()`** — ทำงานคู่กันเมื่อ type/source/mode เปลี่ยน
   - `toggleOutFields()` ถูกเรียกก่อน `renderPinRows()` — ต้องไม่ล้ม (null guard)
   - `renderPinRows()` save/restore pattern — ต้อง cover ทุก field

3. **`OutputControl::loop()`** — วนทุก output channel ทุก frame:
   - LED → NeoPixelBus.Show()
   - DMX → esp_dmx_send()
   - Stepper → move / runSpeed
   - AC Dimmer → timer pulse
   - Virtual 7-Segment → updateDimmer7Segs() (ต้องมาก่อน DMX frame)
   - I2C expander writes → mutex guard
   - **ลำดับสำคัญ:** Virtual 7-Segment update ต้อง happen **before** DMX frame output มิฉะนั้นจะได้ค่าเก่า

4. **`networkTask()`** — รันบน core 0 ทุก 10ms:
   - AsyncWebServer handle
   - Display update (~2Hz)
   - Ethernet loop
   - OTA in-progress check
   - **ต้องไม่ block > 50ms** มิฉะนั้น Web UI จะ timeout

  - Stepper 32-bit position test ยังใช้ UI slider แบบ 8-bit ไม่เหมาะกับค่าพัลส์หลายหมื่น
- **แนวทาง:** ตรวจสอบฟังก์ชัน `testOutput()` ใน main.cpp และ UI ใน `showOutputTest()` ของ web_pages.h:
  - LED: ทดสอบสี RGB/W ผ่าน NeoPixelBus
  - Stepper: รองรับ 32-bit position input
  - Virtual 7-Segment: ส่ง test character ไปยัง API `/api/output-test`
  - AC Dimmer: ทดสอบ PWM level
  - DC Motor: ทดสอบ PWM + DIR/IN pins
- **อ้างอิง:** v1.12.0 section 6.2 (Output Test Menu Fixes)

### 12.2 7-Segment Direct Drive — ต้องการให้ตั้ง GPIO Pin ได้อิสระ
- **ปัญหาปัจจุบัน:** 7-Segment direct drive (mode 2/3, 7-pin/8-pin) ใช้ expander channels เรียงต่อเนื่อง (base channel → base+6/base+7) ไม่สามารถกำหนด GPIO pin แต่ละ segment ได้อิสระ
- **ความต้องการ:** ให้ตั้ง pin ของ segment A-G/DP ได้เองทีละ pin (ไม่ว่าจะเป็น ESP32 GPIO หรือ expander channel ใดก็ได้)
- **แนวทาง:**
  - เพิ่ม field seg_a_pin, seg_b_pin, ..., seg_g_pin, seg_dp_pin (หรือ pin array 7-8 ตัว)
  - แต่ละ segment pin มี source/address/channel ของตัวเอง
  - หรือใช้รูปแบบเดียวกับ pin2/pin3/pin4 แต่เพิ่ม Pin 5-12
  - UI ต้อง expand ได้ 7-8 pin rows
- **ข้อจำกัดเดิม:** v1.13.0 section 15.5 — "รองรับ 1 digit ต่อ output channel"

### 12.3 Virtual 7-Segment — ทำไมไม่พัฒนาต่อเป็น AC Dimmer 7-Segment โดยตรง
- **คำถาม:** เหตุใด Virtual 7-Segment (Type 14) ไม่ถูกพัฒนาต่อเป็น "AC Dimmer 7-Segment" จริงๆ ที่ควบคุมไฟ AC dimmer โดยตรง
- **คำตอบจากสถาปัตยกรรมปัจจุบัน:**
  - Virtual 7-Segment เป็น **virtual output channel** — มันไม่ได้ขับ hardware โดยตรง แต่อ่านค่า ASCII 1 byte จาก DMX buffer → แปลงเป็น segment bits → **เขียนค่าทับไปยัง output channels จริง (type 3, 2, 5 ฯลฯ)** ที่ user ตั้งไว้
  - ถ้าต้องการ AC Dimmer 7-Segment จริงๆ: แค่สร้าง output type 3 (AC Dimmer) จำนวน 7 ช่อง + type 14 (Virtual 7-Segment) 1 ช่องที่ DMX address เดียวกัน → firmware จะ map segment bits ไปยัง dimmer channels อัตโนมัติ
  - **เหตุผลที่แยก:** ทำให้ Virtual 7-Segment ใช้กับ **output type ใดก็ได้** (Relay, PWM Dimmer, DC Motor, etc.) ไม่จำกัดแค่ AC Dimmer
- **ข้อควรระวัง:** Type 14 อ้างอิง output channels ด้วย index — ถ้า user ลบ/เพิ่ม channel ที่อยู่ก่อนหน้า index จะเปลี่ยน → segment mapping ผิด
  - **ยังไม่มี validation** ใน UI/API สำหรับ index stability

### 12.4 ทำความสะอาดโค้ดทั้งหมด (Code Cleanup)
- **Problem:** codebase สะสม technical debt จากการพัฒนาแบบ iterative หลายรอบ:
  - `web_pages.h` — embedded JS กระจัดกระจาย ไม่มี standard pattern
  - `output_control.h` — monolithic header ทุกอย่างในที่เดียว
  - `main.cpp` — route definitions + validation logic + global state ทั้งหมดในไฟล์เดียว
  - Comment/Legacy code — บางส่วนจาก v1.10-v1.12 ที่อาจไม่เกี่ยวข้องแล้ว
- **สิ่งที่ควรทำ:**
  1. แยก route handlers ออกจาก `main.cpp` → `web_routes.h` หรือ `api_handlers.h`
  2. จัดกลุ่ม JS functions ใน `web_pages.h` เป็นหมวดหมู่: `// === Outputs ===`, `// === Settings ===` ฯลฯ
  3. ลบ `#include` ที่ไม่ใช้แล้ว
  4. ตรวจสอบ warning ตอน compile (`-Wall`) และแก้ทั้งหมด
  5. เปลี่ยน `String` concatenation ใน loop เป็น `reserve()` หรือ `printf()` เพื่อลด heap fragmentation
  6. standardize error response format จาก API (ตอนนี้บาง endpoint ส่ง plain text, บาง endpoint ส่ง JSON)

### 12.5 Web UI — Optimize for Controller Resources, Not Aesthetics
- **เป้าหมาย:** ทำให้หน้าเว็บใช้ทรัพยากรคอนโทรเลอร์ (RAM/CPU/Flash) น้อยที่สุด ไม่ต้องสนความสวยงาม
- **แนวทาง:**
  1. **ลด inline CSS** — ใช้ class-based CSS ซ้ำกัน (ตอนนี้หลาย element มี `style="..."` ซ้ำๆ)
  2. **ลด JS size** — รวมฟังก์ชันที่คล้ายกัน (เช่น `deviceLabel()`, `gpioLabel()`, `channelScore()` มี pattern ซ้ำ → สร้าง lookup table)
  3. **Minify HTML/CSS/JS** — อาจใช้ build script ตอน compile (แต่จะอ่านยากตอน debug)
  4. **ลด DOM nodes** — ใช้ `<template>` หรือ document fragment แทน innerHTML (แต่ต้อง balance กับ Flash usage)
  5. **Lazy render** — outputs tab ไม่ต้อง render ทุกครั้งที่ switch tab ถ้า data ไม่เปลี่ยน
  6. **ลด HTTP requests** — รวม API calls (ตอนนี้ save settings → POST /api/settings + POST /api/outputs แยกกัน)
  7. **Avoid `eval()`/`new Function()`** — ใช้ switch-case หรือ object map แทน
  8. **Prefetch** — โหลด `/api/settings` และ `/api/outputs` parallel ตอน page load แทน sequential
  9. **ลด Library dependencies** — ตรวจสอบ platformio.ini ว่า library ไหนใช้จริง บางตัวอาจ unused ตั้งแต่ตัดฟีเจอร์ไป
- **หลักการ:** UX ตอบสนองเร็ว (ไม่กระตุก) + Flash footprint เล็ก = priority > UI สวยงาม

## 13. Hardware Safety: Floating Pin Protection เมื่อถอดบอร์ด Controller

### 13.1 ปัญหา
- **สถานการณ์:** WT32-ETH01 (คอนโทรลเลอร์) ถูกเสียบกับ Interface Board (เช่น บอร์ด I/O, dimmer board, relay board) เวลาถอดคอนโทรลเลอร์ออกมาเพื่ออัปโหลดโปรแกรมใหม่หรือเปลี่ยนบอร์ด IO ทุกขาของ ESP32 จะลอย (floating)
- **ผลกระทบ:**
  - GPIO ที่ลอย → logic level ไม่แน่นอน → MOSFET/Transistor อาจ导通บางส่วน → Load ค้าง/กระพริบ
  - Relay driver → relay อาจกึกกัก (chattering) หรือติดค้าง
  - Stepper driver → EN pin ลอย → motor ค้างในสถานะ energized → ร้อน
  - AC Dimmer (Triac) → Gate ลอย → ไฟติดไม่เต็มที่หรือหลอน
  - I2C bus (SDA/SCL) → ขณะที่ expander ยังมีไฟเลี้ยง → bus contention หรือ latch-up
- **ความเสี่ยง:** **HIGH** — อาจทำให้อุปกรณ์ที่ต่อพ่วงเสียหาย (Overheat, เปิดค้าง, กระชากกระแส)

### 13.2 แนวทางแก้ไข (Hardware Level — แนะนำ)

| วิธี | รายละเอียด | ต้นทุน | ประสิทธิภาพ |
|:---:|---|---|:---:|
| **Pull-down/Pull-up Resistors** | ใส่ resistor 10kΩ ทุก GPIO output (pull-down สำหรับ active-high, pull-up สำหรับ active-low) | ต่ำ (~2 บาท/ตัว) | **สูง** — กัน floating ได้ 100% |
| **Bus Buffer/Isolator** | ใช้ 74HC125 (tri-state buffer) หรือ ISO1541 (I2C isolator) คั่นระหว่าง ESP GPIO และ Interface board — enable pin ต่อกับ ESP GPIO หรือ RC circuit ที่ disable ขณะ boot | กลาง | สูง — แยกวงจรสมบูรณ์ |
| **Series Resistor (330Ω-1kΩ)** | ใส่ resistor อนุกรมทุก GPIO output เพื่อ limit current กรณี floating หรือ short | ต่ำมาก | กลาง — ป้องกัน damage แต่ไม่กัน floating โดยตรง |
| **FET/Transistor คุม Gate ด้วย Pull-down** | ถ้าขับ MOSFET ต่อ resistor 10kΩ จาก Gate→Source (หรือ Gate→GND) | ต่ำ | สูง — MOSFET ปิดอยู่เสมอเมื่อ ESP floating |

### 13.3 แนวทางแก้ไข (Firmware Level — ช่วยได้บางส่วน)

| วิธี | รายละเอียด | ข้อจำกัด |
|:---:|---|---|
| **Pin state restore ใน `setup()`** | ตั้ง GPIO mode + state ทันทีที่ boot (`pinMode(x, OUTPUT); digitalWrite(x, LOW);`) | ก่อนถึง `setup()` ESP32 bootloader (~100ms) ยัง floating |
| **GPIO Pad Hold** | ใช้ `gpio_hold_en()` (RTC GPIO) เพื่อคงสถานะ pin ข้าม reboot | เฉพาะ GPIO 0-39 บางตัว, ต้อง battery backup |
| **Internal Pull-down** | ESP32 มี internal pull-down (~45kΩ) — `pinMode(x, INPUT_PULLDOWN)` แล้วค่อยเปลี่ยนเป็น OUTPUT ตอน init | แรงดึงอ่อนมาก, noise อาจ override ได้ |
| **Startup sequence optimization** | ตั้ง GPIO ให้เป็น safe state โดยเร็วที่สุดใน `setup()`: output + LOW ก่อน init I2C, ก่อน init expander | ไม่แก้ช่วง bootloader |

### 13.4 แนวทางปฏิบัติแนะนำ (Recommended Approach)

รวม hardware + firmware เพื่อ safety สูงสุด:

1. **Hardware:** ใส่ pull-down resistor 10kΩ ทุก GPIO output ที่ไปขับ MOSFET/Relay/Driver
   - โดยเฉพาะ: STEP/DIR/EN ของ stepper, PWM ของ dimmer, Data ของ LED strip, Trig ของ Triac
   - I2C SDA/SCL: pull-up 4.7kΩ (ปกติมีอยู่แล้ว — ถ้าบอร์ด interface ไม่มี ต้องเพิ่ม)
2. **Firmware:** ใน `setup()` ก่อนทุกอย่าง:
   ```cpp
   void setup() {
       // Step 1: Set ALL used GPIO to safe state immediately
       pinMode(4, OUTPUT); digitalWrite(4, LOW);  // LED strip data
       pinMode(5, OUTPUT); digitalWrite(5, LOW);  // STEP
       // ... all pins used in outputs
       // Step 2: Then init I2C, expanders, etc.
       Wire.begin(...);
       // Step 3: Init outputs
       outputCtrl.begin();
   }
   ```
3. **Documentation:** ในคู่มือติดตั้งระบุว่า "ต้องมี pull-down resistor บน GPIO output ทุกตัวที่ออกไปยัง interface board"

### 13.5 GPIO ที่ต้องระวังเป็นพิเศษ

| GPIO | อันตรายถ้าลอย | Recommended Protection |
|:---:|---|---|
| GPIO 4 (LED Data) | LED strip ดวงแรกติดสว่างค้าง | Pull-down 10kΩ + series 330Ω |
| GPIO 5, 18, 19, 21 (STEP pins) | Stepper กระตุกหรือเดินไม่หยุด | Pull-down 10kΩ |
| GPIO 12 (MTDI) | Flash voltage misdetect → ไม่บูต | **หลีกเลี่ยงใช้** หรือ pull-down 10kΩ |
| GPIO 14 (SDA), 15 (SCL) | I2C bus lock | Pull-up 4.7kΩ (จำเป็นอยู่แล้ว) |
| GPIO 32, 33 (ZC pin) | Interrupt storm → controller freeze | Pull-up/down 10kΩ ตาม logic |
| GPIO 25, 26 (DAC/ADC) | PWM output ลอย → MOSFET ค้าง | Pull-down 10kΩ |
| GPIO 13, 2 | On-board LED อาจติดสว่าง | Pull-down 10kΩ หรือ ignore ถ้าไม่รบกวน |

## 16. SPI Expander + AC Dimmer Zero-Crossing Detection

### 16.1 คำถาม
ถ้าใช้บอร์ด expander แบบ SPI (เช่น MCP23S17) สามารถใช้ detect zero-crossing สำหรับ AC dimmer ได้หรือไม่?

### 16.2 คำตอบทางเทคนิค
**ไม่แนะนำ — SPI expander ไม่สามารถใช้สำหรับ ZC detection ได้อย่างมีประสิทธิภาพ**

### 16.3 เหตุผล

| ปัจจัย | รายละเอียด |
|---|---|
| **ZC timing precision** | AC dimmer ต้องการตรวจจับ ZC (±ไม่กี่ μs) เพื่อเริ่ม timer นับ pulse width ถ้าพลาด timing → dimmer pulse ผิด -> ไฟกระพริบ |
| **SPI transaction latency** | MCP23S17 มี INT pin ที่ active เมื่อ input เปลี่ยน แต่ firmware ต้องอ่าน register ผ่าน SPI เพื่อรู้ว่าขาไหนเปลี่ยน → SPI transaction ใช้เวลา ~5-20μs → jitter มากเกินไปสำหรับ dimming |
| **ISR safety** | การทำ SPI transaction ใน ISR context ไม่ปลอดภัย — SPI ใช้ DMA/interrupts ที่อาจ deadlock |
| **GPIO ยังต้องใช้อยู่ดี** | ต้องใช้ ESP32 GPIO 1 ขาเพื่อรับ INT จาก expander อยู่แล้ว → ต่อ ZC ตรงๆ ก็ใช้ 1 GPIO เหมือนกัน |

### 16.4 ข้อสรุป
- ZC pin ต้องต่อ **ตรงกับ ESP32 GPIO interrupt-capable pin** เท่านั้น
- SPI expander (MCP23S17) ใช้กับงาน digital I/O ความเร็วต่ำ (Relay, Solenoid, Smoke, Status LED) ได้ — แต่ **ไม่เหมาะกับ ZC**
- ถ้าต้องการ ZC หลายช่อง (หลาย dimmer board) → ใช้ ESP32 GPIO 1 pin ต่อ ZC 1 ชุด หรือใช้ optocoupler รวม ZC ทุก board เข้าด้วยกัน
- **Recommended:** ต่อ ZC กับ GPIO 32 หรือ 33 (RTC GPIO, interrupt capable) โดยตรง

### 16.5 เปรียบเทียบ I2C Expander vs SPI Expander

| ความสามารถ | I2C (MCP23017) | SPI (MCP23S17) | ใช้กับงานไหนได้ |
|---|---|---|---|
| Digital output (Relay, LED) | ✅ Latency ~200μs | ✅ Latency ~10μs | Relay, Solenoid, Smoke |
| PWM output | ❌ ไม่มี HW PWM | ❌ ไม่มี HW PWM | — |
| Interrupt input | ✅ INT pin | ✅ INT pin | Limit switch, Button |
| ZC detection | ❌ | ❌ | **ต้องใช้ ESP32 GPIO โดยตรง** |
| STEP pulse | ❌ | ❌ | **ต้องใช้ ESP32 GPIO โดยตรง** |
| LED strip data | ❌ | ❌ | **ต้องใช้ ESP32 GPIO โดยตรง** |
| DMX UART | ❌ | ❌ | **ต้องใช้ ESP32 โดยตรง** |

## 18. Feature: DFPlayer MP3 Module (Type 15) — Implemented ✅

### 18.1 Status
**Implemented and OTA uploaded** (v1.15.0 session).  
See files: `include/dfplayer_control.h` (driver), `include/output_control.h` (struct field + init), `include/motion_control.h` (update loop), `include/scoring.h` (score + name), `include/web_pages.h` (UI), `src/main.cpp` (validation).

### 18.2 DMX Control Scheme (Implemented)
3 DMX bytes ต่อ DFPlayer channel:

| Byte | Function | Range | Details |
|:---:|---|---|---|
| 0 | Track | 0-255 | 0=stop, 1-255=play track |
| 1 | Volume | 0-255 | mapped → DFPlayer 0-30 via `map(vol,0,255,0,30)` |
| 2 | Control | 0-6 | 0=normal, 1=next, 2=prev, 3=pause, 4=resume, 5=loop single, 6=loop all |

### 18.3 Driver Design (`dfplayer_control.h`)
- **Class:** `DFPlayerController` (per-channel, ใช้ pointer `DFPlayerController* dfPlayer` ใน `OutputChannel`)
- **Protocol:** raw byte packet (0x7E, 0xFF, 0x06, cmd, paramH, paramL, checksumH, checksumL, 0x00, 0xEF)
- **Serial:** Uses `HardwareSerial` (Serial2, begin at 9600 baud)
- **Debounce:** 50ms minimum gap between commands (via `lastCmdTime`)
- **State tracking:** `lastTrack`, `lastVolume` (uint16) — skip redundant commands
- **Commands implemented:** playTrack, setVolume, stop, pause, resume, next, previous, loopSingle, loopAll
- **Init sequence:** RESET → PLAY_SRC(SD) → VOL(15) with 600ms+300ms delays
- **Method `update(track, volume, control)`:** called from `motion_control.h` every loop — control commands have priority, then volume, then track

## 19. Hardware Resource Conflict Matrix

### 19.1 ทำไมต้องมี
ESP32 มี peripheral จำกัด — output type ต่างๆ แข่งกันใช้ resource เดียวกัน ถ้าไม่ตรวจสอบจะเกิด:

- PWM สั่น/ไม่มี output (LEDC channel ซ้ำ)
- LED strip ไม่แสดงผล (RMT channel เต็ม)
- ESP32 crash/reboot (timer/IRQ conflict)
- AC Dimmer เฟส shift (timer ผิด)

### 19.2 Resource Pool Limits

```
9 Solenoid   -      -      -     -      1      -     
10 Analog RGB -     3-4ch  -     -      1?     -     LEDC 3-4 ch (R/G/B/W); pin 1 แบบ expander

15 DFPlayer  1      -      -     -      2      -     UART2 (TX+RX = 2 pins)
```

### 19.4 Known Conflict Scenarios

#### 🔴 P0: DMX + DFPlayer บน UART2 (Critical)
```
DMX channel 1 ──→ DMX_NUM_2 (UART2)
DFPlayer      ──→ Serial2 (UART2)
→ CRASH / both fail
```
- **Root cause:** DMX `setupChannels()` assigned `DMX_NUM_2` (UART2) first, DFPlayer also uses `Serial2`
- **Fix:** ต้องเลือก UART ให้ DFPlayer แบบ dynamic: ถ้า UART2 ถูก DMX ใช้ → ใช้ UART1 (ถ้า UART1 ถูกใช้ → แจ้ง error)
- **Current code:** `setupChannels()` ทำ DMX ก่อน DFPlayer (DMX ได้ UART2 ก่อน), DFPlayer เสมอใช้ `Serial2` → **ถ้ามี type 1 + type 15 พร้อมกัน จะพัง**
- **Validation gap:** `validateOutputJson()` ไม่ได้ check UART conflict

#### 🟡 P1: LEDC Channel เต็ม (Resource Exhaustion)
```
PWM Dimmer×4 ──→ 8 LEDC channels (2ch × 4)
→ หมด LEDC → output เงียบ
```
- 8 LEDC channels maximum
- Type 5 uses 1-2, type 6 uses 1-3, type 8/11 uses 1, type 10 uses 3-4
- **Current:** ไม่มี validation ตรวจ LEDC resource pool

#### 🟡 P1: RMT Channel เต็ม
```
LED Strips ×8 ──→ 8 RMT channels
→ Over limit → สร้าง strip ไม่ได้
```
- `setupChannels()` มี check `if (rmtIdx < 8)` — จำกัดไว้แล้ว
- แต่ DMX fallback ก็ใช้ RMT → ต้อง tracking rmtIdx ให้ถูกต้อง

#### 🟢 P2: GPIO Pin Conflicts
- WT32-ETH01 มี usable GPIO จำกัด (~8-12)
- Type 6/7/13 ใช้หลาย pins → อาจ conflict
- **Current:** มี `outputsHaveDuplicateGpio()` validation — detect pin ซ้ำ

#### 🟢 P2: I2C Address Conflicts
- PCA9685 ใช้ 0x40-0x47, MCP23017 ใช้ 0x20-0x27
- ถ้า user ตั้ง address ซ้ำ → I2C device ล้มเหลว
- **Current:** ไม่มี validation ตรวจ I2C address ซ้ำระหว่าง output channels (มีแค่ expander channel ซ้ำ)

### 19.5 Priority Action Items

| Priority | Issue | Action |
|:--------:|-------|--------|
| **P0** | DMX + DFPlayer UART2 conflict | `setupChannels()` ต้อง reserve UART dynamic; หรือให้ DFPlayer ใช้ Serial1 ถ้า Serial2 ถูกใช้ |
| **P1** | LEDC allocation tracking | สร้าง `ledcAllocator` นับ channel ที่ใช้แล้ว ป้องกัน overcommit |
| **P2** | I2C address conflict check | เพิ่ม `outputsHaveDuplicateI2cAddr()` validation |
| **P3** | Document resource limits | Matrix นี้ต้อง update ทุกครั้งที่มี type ใหม่ |

### 19.6 UART Allocation Strategy (Recommendation)

ควรเพิ่ม UART manager ใน output_control.h:

```cpp
class UartAllocator {
    bool uart1Used = false;  // DMX_NUM_1
    bool uart2Used = false;  // DMX_NUM_2
public:
    bool allocate(uint8_t& outPort) {
        if (!uart2Used) { outPort = 2; uart2Used = true; return true; }
        if (!uart1Used) { outPort = 1; uart1Used = true; return true; }
        return false; // No free UART
    }
    void free(uint8_t port) {
        if (port == 1) uart1Used = false;
        if (port == 2) uart2Used = false;
    }
};
```

แล้ว DMX และ DFPlayer เรียก `uartAllocator.allocate()` แทน hardcode `DMX_NUM_2` / `Serial2`  
DFPlayer ใช้ Serial1 ถ้า UART2 ถูก DMX จับไปแล้ว, หรือใช้ Serial2 ถ้า DMX ไม่ได้ใช้ UART2  

### 19.7 Summary
Hardware conflict เป็น **P0-P1** ที่ต้องแก้ก่อน production.  
จุดสำคัญที่สุดคือ **UART2 conflict ระหว่าง DMX กับ DFPlayer** — เกิดทันทีเมื่อ user มี type 1 + type 15 พร้อมกัน.  
นอกจากนี้ LEDC/RMT pool tracking เป็น P1 ที่ควรเพิ่ม validation.
